## 什么时候审查代码

前面所有功能实现完并通过测试后，进入代码审查阶段。Superpowers 有两个与代码审查相关的 Skill：`requesting-code-review`（请求审查）和 `receiving-code-review`（接收审查反馈）。

`requesting-code-review` 需要手动调用，可以在以下时间点执行：

- **主要功能实现完成后** — 整体验证代码质量
- **合并到主分支前** — 最后一道质量关
- **遇到困难时** — 换个视角审视代码，可能发现盲点
- **重构前** — 先确认当前代码的基线状态
- **复杂 bug 修复后** — 确认修复没有引入新问题

注意表格里没有"子代理驱动开发中每个 Task 完成后"。因为那个场景有自己专用的审查流程，不走 `requesting-code-review`。下面详细说明。

### subagent-driven-development 的内置审查 vs requesting-code-review

`requesting-code-review` 是一个独立的 Skill，手动调用。而 `subagent-driven-development` 有自己的内置审查流程，不依赖 `requesting-code-review`。两者的设计思路不同。

**subagent-driven-development 的内置审查：两阶段、自动、per-Task**

在子代理驱动开发中，每完成一个 Task，编排代理会自动派发**两个**审查子代理，依次执行：

1. **spec reviewer**（`spec-reviewer-prompt.md`）— 检查代码是否符合规格。比如规格要求"验证码 6 位数字"，代码里写的是 4 位，spec reviewer 会报错。它只关心"做了没有"，不关心"做得好不好"。
2. **code quality reviewer**（`code-quality-reviewer-prompt.md`）— 在 spec 通过后，检查代码质量。比如命名是否清晰、有没有重复逻辑、测试覆盖是否充分。

为什么分成两个阶段？因为审查目的不同：

- spec reviewer 回答的是"**做对了没有**"——功能是否完整、有没有多做或少做
- code quality reviewer 回答的是"**做好了没有**"——实现是否优雅、可维护性如何

分开的好处是各自聚焦，避免一个审查者同时关注两件事导致遗漏。`subagent-driven-development` 明确要求：**spec reviewer 不通过，不能进入 code quality review**。如果规格都没对齐，讨论代码质量没有意义。

用 auth API 举例。假设 Task 2 实现了 `POST /auth/verify`，完成后的审查流程：

```
Task 2 完成
  → spec reviewer：检查 verify 接口是否做了规格要求的所有事情
    - 验证码校验？✅
    - 自动注册新用户？❌ 漏了
    - 自动登录已有用户？✅
    - 签发 JWT？✅
    → 不通过 → 实现子代理补上自动注册 → 重新 spec review → 通过

  → code quality reviewer：检查实现质量
    - 测试覆盖？✅
    - 错误处理？✅
    - 验证码比较用了 equals 而不是恒等比较？⚠️ 安全问题
    → 不通过 → 实现子代理修复 → 重新 code quality review → 通过

Task 2 完成
```

**requesting-code-review：单次、手动、任意范围**

`requesting-code-review` 是一个独立的审查入口，你在需要的时候手动调用。它派发一个审查子代理，用 git SHA 定义审查范围，一次审查同时关注规格合规和代码质量。

两者的核心区别：

| 维度 | 内置审查（subagent-driven-development） | requesting-code-review |
|------|---------------------------------------|----------------------|
| 审查阶段 | 两阶段：先 spec 再 quality | 单次审查，规格+质量一起看 |
| 触发方式 | 自动，每个 Task 完成后 | 手动调用 |
| 审查范围 | 单个 Task 的 diff | 任意范围，用 git SHA 指定 |
| 使用的模板 | `spec-reviewer-prompt.md` + `code-quality-reviewer-prompt.md` | `requesting-code-review` 自己的模板 |
| 适用场景 | 子代理驱动开发流程内 | 流程外的独立审查 |

简单说：内置审查发生在每个 Task 完成后，重点检查当前 Task 的实现质量；`requesting-code-review` 可以在任意时间触发，检查你指定范围内的代码。

## 请求审查

`requesting-code-review` 只负责审查，不负责修改代码或提交 PR。整个流程分三步：

1. **你指定审查范围**（用 git SHA）
2. **审查子代理返回反馈**（Strengths + Issues）
3. **你来处理反馈**（修改代码、反驳、或记录）

修改代码是你或实现子代理的事，提交 PR 是 `finishing-a-development-branch` 的事。

### 基于 Git SHA 的审查

requesting-code-review 使用 git SHA 来定义审查范围：

先用一句话解释什么是 git SHA。Git 中每一次提交（commit）都有一个唯一标识，长这样：

```
a7981ec3df7661b2c4d5e6f7a8b9c0d1e2f3a4b5
```

这串字符就是 SHA（Secure Hash Algorithm），由 Git 根据提交内容自动计算生成。通常只取前 7 位就够用了，比如 `a7981ec`。SHA 和提交是一对一的关系——改了任何一个字符，SHA 就变了。

审查时用两个 SHA 圈定范围。SKILL.md 中的默认写法是：

```bash
BASE_SHA=$(git rev-parse HEAD~1)  # or origin/main
HEAD_SHA=$(git rev-parse HEAD)
```

默认 `HEAD~1`，即**只看最近一次提交**。注释中的 `origin/main` 是替代选项，表示"从主分支分叉以来的所有变更"。示例中还展示了第三种写法——通过 `git log` 搜索特定 commit 作为起点，比如上一个 Task 的提交。起点怎么选取决于审查目的，但如果你不特别指定，就是只审最新一次提交。

然后拿这两个 SHA 派发审查子代理。子代理看到的就是从 `a7981ec` 到 `3df7661` 之间的 diff，不会看到无关的代码变更。

### 处理审查反馈

审查子代理返回的反馈会按严重程度分级。原版规则：

> - Fix Critical issues immediately
> - Fix Important issues before proceeding
> - Note Minor issues for later
> - Push back if reviewer is wrong (with reasoning)

即：Critical 立即修复、Important 继续下一个任务前修复、Minor 记录以后处理、审查者错了就用技术理由反驳。

用 auth API 举例。假设整体审查子代理返回：

```
Strengths: 架构清晰，测试覆盖良好，API 设计符合 RESTful 规范

Issues:
  Important: 手机号格式校验在 Controller 和 Service 中重复实现，
            应统一到 Service 层
  Minor: JwtService.generateToken() 方法中的过期时间硬编码为 3600，
        建议提取为配置项
```

Important 级别的问题必须修。修完后重新运行测试确认没有引入新问题。Minor 级别的记录下来，不阻断当前流程。

### 审查者可能是错的

requesting-code-review 明确说：**如果审查者错了，用技术理由反驳**。

> Push back with technical reasoning. Show code/tests that prove it works.

审查者的建议是"建议"，不是"命令"。如果审查者说"这里应该用 JWT 的 RS256 算法"，但你的设计文档明确说用 HS256（对称加密），而且代码中有测试证明 HS256 工作正常——就可以反驳。

## 接收审查反馈

先理清两个 Skill 的关系。`requesting-code-review` 发起审查后，审查子代理把反馈返回给当前会话，不涉及 GitHub PR。而 `receiving-code-review` 定义的是**收到反馈后怎么处理**，适用于任何来源的审查反馈：

- `requesting-code-review` 触发的审查子代理返回的反馈
- 子代理驱动开发中 spec reviewer / code quality reviewer 返回的反馈
- 用户直接告诉你的反馈（"修复第 1-6 项"）
- GitHub PR 上的评论

整个流程是：**发起审查 → 拿到反馈 → 验证反馈是否正确 → 正确则修复，错误则反驳，不清楚则先问。** 前半段是 `requesting-code-review` 的事，后半段是 `receiving-code-review` 的事。

`receiving-code-review` 不会自动拉取 PR 或获取反馈，它只定义了一件事：**反馈已经在手上了，怎么处理才是对的？**

### 禁止的回复

Skill 列出了禁止的回复和正确做法，原版如下：

```
NEVER:
- "You're absolutely right!" (explicit CLAUDE.md violation)
- "Great point!" / "Excellent feedback!" (performative)
- "Let me implement that now" (before verification)

INSTEAD:
- Restate the technical requirement
- Ask clarifying questions
- Push back with technical reasoning if wrong
- Just start working (actions > words)
```

禁止的回复都是**表演性同意（performative）**——表达的是"我在情感上认可你"，而不是"我在技术上理解了这个问题"。

Skill 的要求是：

> Actions speak. Just fix it. The code itself shows you heard the feedback.

收到正确的反馈时，正确做法是直接修复，然后用一句话说明改了什么：

```
✅ "Fixed. 手机号校验统一到 Service 层。"
✅ "Good catch — 手机号校验逻辑重复。已合并到 AuthService。"
❌ "你说得太对了！让我立刻修改这个非常重要的问题！"
```

### YAGNI 检查

receiving-code-review 有一个特别实用的模式——YAGNI 检查。当审查者建议"实现一个完整的功能"时，先检查这个功能有没有人用：

```
审查者: "建议实现完整的 metrics 跟踪，包含数据库存储、日期过滤、CSV 导出"

你: grep 代码库 → 没有任何地方调用 metrics 端点

你: "代码库中没有调用 metrics 端点。按 YAGNI 原则，移除它还是你确实需要？"
```

Skill 引用了用户的规则："你和审查者都向我汇报。如果我们不需要这个功能，就不要加。"

这条规则的存在是因为 AI 有一个倾向：审查者说"应该加 X"，实现者就不假思索地加。但很多时候审查者（尤其是 AI 审查者）建议的功能是"理论上好的"而不是"实际需要的"。YAGNI 检查是一个过滤器：先证明需要，再动手实现。

### 不清楚时先问

如果审查反馈有多条，其中一些不清楚：

```
审查者: "修复第 1-6 项问题"

你理解了 1、2、3、6，不清楚 4 和 5。

❌ 错误：先实现 1、2、3、6，稍后再问 4、5
✅ 正确："我理解第 1、2、3、6 项。在实现之前需要澄清第 4 和第 5 项。"
```

为什么？因为各条反馈之间可能有依赖关系。对第 4 条的理解可能影响第 5 条的实现方式。部分理解 = 错误实现。

## finishing-a-development-branch：分支收尾

代码审查完成，所有问题修复后，触发 `finishing-a-development-branch` Skill。

这个 Skill 有 6 个步骤，我们一起走一遍。

### Step 1：验证测试

```
I'm using the finishing-a-development-branch skill to complete this work.
```

第一步永远是运行测试：

```bash
mvn test
```

如果测试失败，停下来，不继续。Skill 说：

> Tests failing (<N> failures). Must fix before completing. Cannot proceed with merge/PR until tests pass.

测试通过才进入 Step 2。

### Step 2：检测环境

```bash
GIT_DIR=$(cd "$(git rev-parse --git-dir)" 2>/dev/null && pwd -P)
GIT_COMMON=$(cd "$(git rev-parse --git-common-dir)" 2>/dev/null && pwd -P)
```

这段命令检测当前是在普通 git 仓库中，还是在 git worktree 中。检测结果决定了后续的菜单选项和清理行为：

- **普通仓库**（`GIT_DIR == GIT_COMMON`）：展示标准 4 选项（合并、PR、保留、丢弃），没有 worktree 需要清理。
- **Git Worktree + 有名分支**：同样展示标准 4 选项。完成后根据 worktree 的创建来源决定是否清理——如果是 Superpowers 创建的 worktree，会自动清理；如果是你通过 IDE 或其他工具创建的，不碰。
- **Git Worktree + detached HEAD**（无分支名）：只展示 3 选项，没有"合并到主分支"这个选项（因为 detached HEAD 没有分支名可以合并）。不清理 worktree，由外部管理。

### Step 3：确定基准分支

找到当前分支是从哪个分支切出来的（通常是 main 或 master）。

### Step 4：展示选项

```
Implementation complete. What would you like to do?

1. Merge back to main locally
2. Push and create a Pull Request
3. Keep the branch as-is (I'll handle it later)
4. Discard this work

Which option?
```

我们选择方案 2：推送并创建 PR。

### Step 5：执行选择

```bash
# 推送分支
git push -u origin feature/auth-api

# 创建 PR
gh pr create --title "feat: add auth API with phone verification" --body "$(cat <<'EOF'
## Summary
- POST /auth/send-code：发送验证码接口
- POST /auth/verify：验证码校验 + 自动注册/登录 + JWT 签发
- GET /auth/me：获取当前用户信息（需 Token）
- 全局错误处理和参数校验

## Test Plan
- [ ] 验证发送验证码接口（正常 + 无效手机号）
- [ ] 验证注册新用户流程
- [ ] 验证已有用户登录流程
- [ ] 验证用户信息接口（有效/无效/过期 Token）
EOF
)"
```

注意：选择创建 PR 时，**不清理 worktree**。因为用户可能需要在 PR 反馈后继续迭代代码。

### Step 6：清理 workspace

清理只在选项 1（本地合并）和选项 4（丢弃）时执行。

而且清理有一个重要的**来源检查**：只清理 Superpowers 创建的 worktree（路径在 `.worktrees/`、`worktrees/` 或 `~/.config/superpowers/worktrees/` 下的）。如果是宿主环境创建的 worktree，不碰。

这个设计防止了一个常见错误：用户通过 IDE 或其他工具创建的 worktree，被 Superpowers 误删。

```bash
# 来源检查
if worktree path is under .worktrees/ or worktrees/ or ~/.config/superpowers/worktrees/:
    # Superpowers 创建的，安全清理
    cd "$MAIN_ROOT"
    git worktree remove "$WORKTREE_PATH"
    git worktree prune
else:
    # 宿主环境创建的，不碰
    leave it in place
```

## 核心工作流总结

到这里，Superpowers 的核心工作流已经讲完了。用每个 Skill 一句话总结：

```
brainstorming（头脑风暴）
  → 探索需求、理解意图，输出设计文稿（spec）

writing-plans（编写计划）
  → 拿着设计文稿，拆成可执行的 Task 列表，附上下文和验收标准

subagent-driven-development（子代理驱动开发）
  → 为每个 Task 派发独立子代理实现，内置两阶段审查

using-git-worktrees（使用 Git Worktree）
  → 为开发创建隔离的工作空间，避免污染主分支

test-driven-development（测试驱动开发）
  → 先写失败测试再写实现，RED-GREEN-REFACTOR 循环

systematic-debugging（系统化调试）
  → 遇到 bug 先找根因，不猜不碰运气，3 次失败则审查架构

verification-before-completion（完成前验证）
  → 声称"完成"前必须跑命令、读输出、拿证据

requesting-code-review（请求审查）
  → 手动发起代码审查，用 git SHA 指定范围，返回分级反馈

receiving-code-review（接收审查反馈）
  → 收到任何来源的反馈后，验证、反驳或修复，不做表演性同意

finishing-a-development-branch（分支收尾）
  → 测试通过后提供合并、PR、保留、丢弃四个选项，安全清理 worktree
```
