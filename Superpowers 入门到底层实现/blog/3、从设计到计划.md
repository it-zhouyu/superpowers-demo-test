## 设计批准后发生什么

前面我们用 brainstorming 完成了 auth API 的设计，你批准了设计文档。Claude Code 自动调用了 `writing-plans` Skill。

此时你会看到：

```
I'm using the writing-plans skill to create the implementation plan.
```

Claude Code 开始把设计文档翻译成一份详细的、可直接执行的实现计划。

所谓"执行计划"（Implementation Plan），本质上就是**开发任务列表**。writing-plans 做的事情就是把设计文档拆分成一个一个具体的开发任务，每个任务包含要创建/修改的文件、完整的代码、测试和提交命令。创建执行计划 = 生成开发任务列表。

## Plan 文件的整体结构

一份完整的执行计划由三部分组成：

**1. Plan Header（计划头）**：声明目标、架构、技术栈，告诉执行者"要做什么"。

**2. File Structure（文件列表）**：列出所有要创建、修改和测试的文件，让执行者对全貌有概念。

**3. Tasks（任务列表）**：一个一个具体的开发任务，每个任务包含完整代码、测试命令和预期输出。

对照前面讲到的设计文档，计划文件把"做什么"翻译成了"怎么做"：

| 设计文档 | 计划文件 |
|---------|---------|
| API 接口定义 | 具体的 Controller 代码 |
| 数据结构设计 | 具体的 Model/DTO 类 |
| 错误处理规则 | 具体的异常处理代码 |
| 技术选型 | Header 中的 Tech Stack |
| 验收标准 | 每步的 Expected 输出 |

接下来逐段讲解每个部分。

## Plan Header：计划头

```markdown
# [Feature Name] Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans to implement this plan task-by-task.
> Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** [One sentence describing what this builds]

**Architecture:** [2-3 sentences about approach]

**Tech Stack:** [Key technologies/libraries]

```

模板很简洁——四个必填字段。`[Feature Name]` 替换为功能名称，`[One sentence...]` 替换为一句话目标描述，`[2-3 sentences...]` 替换为架构说明，`[Key technologies...]` 替换为技术栈列表。

注意第一行 `REQUIRED SUB-SKILL`，这是给执行者的指令：必须用 `subagent-driven-development` 或 `executing-plans` 来执行这个计划，不能自己随意实现。

把我们的 auth API 需求套进模板：

```markdown
# Auth API Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans to implement this plan task-by-task.
> Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 实现手机号验证码统一登录注册 REST API

**Architecture:** Spring Boot 3.x 单体应用，ConcurrentHashMap 内存存储，
JWT 无状态认证，三层架构（Controller → Service → Repository）

**Tech Stack:** Java 17, Spring Boot 3.x, Maven, JUnit 5, MockMvc

```

这个 Header 不是装饰——它告诉后续执行的子代理三件事：目标是什么、架构怎么组织、用什么技术栈。Claude Code 的子代理不继承当前会话的上下文，它拿到的是一个全新上下文，看到的第一个内容就是这个 Header。

> 补充：Claude Code 的 `/fork` 命令（即 `/branch`）可以在保留当前会话上下文的情况下创建新的分支会话。

## File Structure：文件列表

在定义具体任务之前，Plan 会先列出所有要创建和修改的文件，以及每个文件的职责：

```markdown
## File Structure

- Create: `src/main/java/com/example/auth/AuthApplication.java` — Spring Boot 启动类
- Create: `src/main/java/com/example/auth/controller/AuthController.java` — 认证接口
- Create: `src/main/java/com/example/auth/service/AuthService.java` — 认证业务逻辑
- Create: `src/main/java/com/example/auth/service/JwtService.java` — JWT 签发与验证
- Create: `src/main/java/com/example/auth/model/User.java` — 用户模型
- Create: `src/main/java/com/example/auth/dto/SendCodeRequest.java` — 发送验证码请求 DTO
- Create: `src/main/java/com/example/auth/dto/VerifyRequest.java` — 验证请求 DTO
- Create: `src/main/java/com/example/auth/dto/ApiResponse.java` — 统一响应 DTO
- Create: `src/main/java/com/example/auth/config/JwtAuthFilter.java` — JWT 认证过滤器
- Create: `src/main/java/com/example/auth/config/SecurityConfig.java` — Spring Security 配置
- Modify: `pom.xml` — 添加 Spring Boot Starter Web、Security、JWT 依赖
- Test: `src/test/java/com/example/auth/controller/AuthControllerTest.java`
- Test: `src/test/java/com/example/auth/service/AuthServiceTest.java`
```

为什么要在任务之前列文件？因为 writing-plans 的设计哲学是：**假设执行者是一个技能不错但对你项目一无所知的工程师**。先列出所有文件，让执行者对全貌有概念，再做具体任务。这比上来就写代码更可靠。

## Tasks：任务列表

writing-plans 有一个关键约束：**每个 step 是一个 2-5 分钟就能完成的原子操作**。

这不是随意定的。一个 step 做太多事情，执行者容易遗漏或犯错。拆得太细，计划膨胀到无法管理。2-5 分钟是经验验证过的平衡点。

先看 SKILL.md 中定义的原始 Task Structure 模板：

````markdown
### Task N: [Component Name]

**Files:**
- Create: `exact/path/to/file.py`
- Modify: `exact/path/to/existing.py:123-145`
- Test: `tests/exact/path/to/test.py`

- [ ] **Step 1: Write the failing test**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **Step 2: Run test to verify it fails**

Run: `pytest tests/path/test.py::test_name -v`
Expected: FAIL with "function not defined"

- [ ] **Step 3: Write minimal implementation**

```python
def function(input):
    return expected
```

- [ ] **Step 4: Run test to verify it passes**

Run: `pytest tests/path/test.py::test_name -v`
Expected: PASS

- [ ] **Step 5: Commit**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

模板的要点：

**1. Files 区块**。每个 Task 开头列出要创建（Create）、修改（Modify）和测试（Test）的文件。`Modify` 还要指定行号范围，让执行者精确定位。

**2. 5 步 TDD 节奏**。写失败测试 → 验证失败 → 写最小实现 → 验证通过 → 提交。每个 Step 用 `- [ ]` checkbox 语法标记，执行者完成后打勾。

**3. 代码是完整的**。Step 1 和 Step 3 里都是完整的可运行代码，不是伪代码或描述。

**4. 命令是具体的**。`pytest tests/path/test.py::test_name -v`，不是"运行测试"。

**5. 有预期输出**。`Expected: FAIL with "function not defined"`、`Expected: PASS`，让执行者知道运行后应该看到什么。

## 套用模板：以 Task 2 为例

上面讲的是通用模板，现在用 auth API 来套。完整的计划有 5 个 Task，这里用 Task 2（发送验证码接口）来举例，其他 Task 的结构完全一样，只是代码内容不同。

模板中 `Task N: [Component Name]` 对应到实际需求就是 `Task 2: POST /auth/send-code 发送验证码`。模板中 `Create: exact/path/to/file.py` 对应到实际要创建的 Java 文件。每一步的代码、命令、预期输出都从模板的占位符变成了具体内容：

````markdown
### Task 2: POST /auth/send-code 发送验证码

**Files:**
- Create: `src/main/java/com/example/auth/controller/AuthController.java`
- Create: `src/main/java/com/example/auth/dto/SendCodeRequest.java`
- Create: `src/main/java/com/example/auth/dto/ApiResponse.java`
- Create: `src/main/java/com/example/auth/service/AuthService.java`
- Test: `src/test/java/com/example/auth/controller/AuthControllerTest.java`

- [ ] **Step 1: 写失败的测试**

```java
@SpringBootTest
@AutoConfigureMockMvc
class AuthControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Test
    void shouldSendCodeToValidPhone() throws Exception {
        mockMvc.perform(post("/auth/send-code")
                .contentType(MediaType.APPLICATION_JSON)
                .content("{\"phone\":\"13800138000\"}"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.message").value("验证码已发送"));
    }

    @Test
    void shouldRejectInvalidPhoneFormat() throws Exception {
        mockMvc.perform(post("/auth/send-code")
                .contentType(MediaType.APPLICATION_JSON)
                .content("{\"phone\":\"123\"}"))
            .andExpect(status().isBadRequest());
    }
}
```

- [ ] **Step 2: 运行测试，确认失败**

Run: `mvn test -Dtest=AuthControllerTest -pl .`
Expected: FAIL — classes not found

- [ ] **Step 3: 写最小实现**

```java
// SendCodeRequest.java
public record SendCodeRequest(
    @Pattern(regexp = "^1[3-9]\\d{9}$", message = "手机号格式无效")
    String phone
) {}

// ApiResponse.java
public record ApiResponse(String message, Object data) {
    public static ApiResponse success(String message) {
        return new ApiResponse(message, null);
    }
}

// AuthService.java
@Service
public class AuthService {
    private final ConcurrentHashMap<String, String> codeStore = new ConcurrentHashMap<>();

    public void sendCode(String phone) {
        codeStore.put(phone, "123456");
    }
}

// AuthController.java
@RestController
@RequestMapping("/auth")
public class AuthController {
    private final AuthService authService;

    public AuthController(AuthService authService) {
        this.authService = authService;
    }

    @PostMapping("/send-code")
    public ResponseEntity<ApiResponse> sendCode(@Valid @RequestBody SendCodeRequest request) {
        authService.sendCode(request.phone());
        return ResponseEntity.ok(ApiResponse.success("验证码已发送"));
    }
}
```

- [ ] **Step 4: 运行测试，确认通过**

Run: `mvn test -Dtest=AuthControllerTest -pl .`
Expected: PASS, 2 tests

- [ ] **Step 5: 提交**

```bash
git add src/main/java/com/example/auth/controller/AuthController.java \
        src/main/java/com/example/auth/dto/SendCodeRequest.java \
        src/main/java/com/example/auth/dto/ApiResponse.java \
        src/main/java/com/example/auth/service/AuthService.java \
        src/test/java/com/example/auth/controller/AuthControllerTest.java
git commit -m "feat: add POST /auth/send-code endpoint"
```
````

对照上面的模板，可以看到 Task 2 完全遵循了 5 步 TDD 节奏：先写测试（Step 1）→ 确认失败（Step 2）→ 写实现（Step 3）→ 确认通过（Step 4）→ 提交（Step 5）。这正是 TDD 的 RED-GREEN-REFACTOR 循环在计划层面的体现。

其他 Task（Task 1 项目初始化、Task 3 验证接口、Task 4 用户信息接口、Task 5 错误处理）的结构和 Task 2 完全一致，都是同样的 5 步节奏，只是代码内容不同。完整的任务列表如下：

| Task | 内容 | 关键实现 |
|------|------|---------|
| Task 1 | Spring Boot 项目初始化 | pom.xml、启动类、基础目录结构 |
| Task 2 | POST /auth/send-code | Controller、Service、DTO、参数校验 |
| Task 3 | POST /auth/verify | 验证码校验、自动注册/登录、JWT 签发 |
| Task 4 | GET /auth/me | JWT 解析过滤器、用户信息返回 |
| Task 5 | 错误处理与边界情况 | 全局异常处理、Token 过期、无效请求 |

## No Placeholders 原则

writing-plans 有一条硬规则叫 **No Placeholders**（不允许占位符）。原版如下：

> Every step must contain the actual content an engineer needs. These are **plan failures** — never write them:
> - "TBD", "TODO", "implement later", "fill in details"
> - "Add appropriate error handling" / "add validation" / "handle edge cases"
> - "Write tests for the above" (without actual test code)
> - "Similar to Task N" (repeat the code — the engineer may be reading tasks out of order)
> - Steps that describe what to do without showing how (code blocks required for code steps)
> - References to types, functions, or methods not defined in any task

简单解读：

- 写了 "TBD" 或 "TODO" → 执行者不知道该做什么
- 写了 "添加适当的错误处理" → 什么是"适当"？必须写出具体代码
- 写了 "为上面的代码写测试" 但没给测试代码 → 执行者可能写得不够
- 写了 "和 Task N 类似" → 执行者可能不按顺序读，看不到 Task N
- 只描述做什么不展示怎么做 → 计划的意义是消除歧义，不是增加歧义

这条规则的核心理念：**计划是给一个对你项目一无所知的人看的**。

## Self-Review：计划自审

计划写完后，writing-plans 要求做一轮自审。这和 brainstorming 的自审类似，但关注点不同：

**1. Spec 覆盖率**：回头去看设计文档，逐项检查——每个需求在计划中都有对应的 Task 吗？有没有设计文档提到了但计划遗漏的功能？

**2. 占位符扫描**：搜索计划中是否有反面模式表里的内容。有就立刻修正。

**3. 类型一致性**：Task 2 里定义的方法叫 `sendCode(String phone)`，Task 3 里引用的是 `sendCode(String phoneNumber)`——名字不一致，执行者会困惑。这种不一致必须修掉。

writing-plans 对自审的态度很务实："发现问题就直接修，不用重新审查。如果发现设计文档有遗漏的需求，加一个 Task。"

## 执行方式选择

自审通过后，计划保存到 `docs/superpowers/plans/2026-06-03-auth-api.md` 并提交 git。Claude Code 会给你两个选择：

```
Plan complete and saved to docs/superpowers/plans/2026-06-03-auth-api.md.
Two execution options:

1. Subagent-Driven (recommended) — I dispatch a fresh subagent per task,
   review between tasks, fast iteration

2. Inline Execution — Execute tasks in this session using executing-plans,
   batch execution with checkpoints

Which approach?
```

**方案 1：子代理驱动（推荐）**。每个 Task 派发一个全新的子代理去执行，当前 Claude Code 会话负责协调和审查。子代理不继承当前会话的上下文，所以每个 Task 必须自包含——这也解释了为什么 Plan 要写得这么详细。优点是子代理上下文干净、可以并行、当前会话保持轻量。

**方案 2：内联执行**。留在当前会话中批量执行任务，设置检查点让你审查。适合不支持子代理的平台。## 为什么需要计划这一步

有人会问：设计文档已经够详细了，为什么不直接让 AI 按设计实现？

writing-plans 存在的意义在于**消除歧义到代码级别**。设计文档说"实现发送验证码接口"，但具体用什么类名、什么目录结构、测试怎么写、命令怎么跑——这些在设计阶段不需要细化，但在实现阶段必须明确。

计划把设计文档的每个模糊点都钉死为具体的代码和命令。这样执行者（无论是子代理还是人类）拿到计划后，不需要做任何判断，按 step 一步步执行就行。

这背后的设计哲学是：**把判断集中在上游（设计和计划），让下游（实现）变成机械执行**。上游做对了，下游就不会出错。
