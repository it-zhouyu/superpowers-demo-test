前面我们让 Agent 学会了调用工具——查天气、做计算，能力越来越强。但有个问题一直被忽略：每次程序一关，对话全没了。下次启动，AI 根本不记得你是谁、聊过什么。

这节来解决记忆问题。

## ChatMessageHistory：把消息存下来

先看最基本的需求——在程序运行期间，把聊天消息保存起来。

```python
from langchain_community.chat_message_histories.in_memory import ChatMessageHistory

# 创建一个内存中的聊天历史记录
history = ChatMessageHistory()

# 添加用户消息和 AI 消息
history.add_user_message("你好，我想学习 Python")
history.add_ai_message("你好！Python 是一门非常适合入门的编程语言。")
history.add_user_message("从哪里开始学比较好？")
history.add_ai_message("建议从基础语法开始，比如变量、循环、函数。")

# 查看所有消息
for msg in history.messages:
    print(f"[{msg.type}] {msg.content}")
```

输出：

```
[human] 你好，我想学习 Python
[ai] 你好！Python 是一门非常适合入门的编程语言。
[human] 从哪里开始学比较好？
[ai] 建议从基础语法开始，比如变量、循环、函数。
```

`ChatMessageHistory` 就是一个内存中的列表容器，能往里面添加消息，也能读出来。它有几个方法：

| 方法 | 说明 |
|------|------|
| `add_user_message(text)` | 添加一条用户消息 |
| `add_ai_message(text)` | 添加一条 AI 消息 |
| `add_message(msg)` | 添加一条任意类型的消息（HumanMessage、AIMessage、SystemMessage 等） |
| `messages` | 属性，返回所有消息的列表 |
| `clear()` | 清空所有消息 |

这里用到了 `ChatMessageHistory`，它是 LangChain 提供的内存消息历史实现。还可以用字典按 session_id 管理多个会话：

```python
store = {}  # session_id -> ChatMessageHistory
store["session_1"] = history
store["session_2"] = ChatMessageHistory()
```

光能存消息还不够，关键是要让 Chain 每次调用时自动把历史消息带进去。

## RunnableWithMessageHistory：让 Chain 自动管理历史

前面我们用 LCEL 拼出了 Chain：`prompt | llm | parser`。现在要在它外面包装一层，让它每次调用时自动带上历史消息。

```python
from langchain_core.runnables import RunnableWithMessageHistory
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

# 带 history 占位符的 prompt
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个友好的编程助手，用中文回答问题。回答要简洁。"),
    MessagesPlaceholder("history"),  # 历史消息会插在这里
    ("human", "{input}"),
])

chain = prompt | llm | StrOutputParser()
```

`MessagesPlaceholder("history")` 是一个占位符，运行时会被实际的历史消息列表替换。这样 prompt 就变成了：system 指令 + 历史对话 + 当前用户输入，正好是 LLM 需要的格式。

接下来定义会话管理函数，然后用 `RunnableWithMessageHistory` 包装 Chain：

```python
# 用字典存储多个会话的历史记录
store = {}

def get_session_history(session_id: str) -> ChatMessageHistory:
    """根据 session_id 获取或创建对应的聊天历史"""
    if session_id not in store:
        store[session_id] = ChatMessageHistory()
    return store[session_id]

# 用 RunnableWithMessageHistory 包装 chain
chain_with_history = RunnableWithMessageHistory(
    chain,
    get_session_history,
    input_messages_key="input",
    history_messages_key="history",
)
```

这段代码做了三件事：

1. `store` 字典按 session_id 存储不同的会话历史
2. `get_session_history` 函数负责查找或创建会话
3. `RunnableWithMessageHistory` 包装 Chain，自动在每次调用前后处理历史消息

现在来看看多轮对话效果：

```python
# 第一轮对话
response1 = chain_with_history.invoke(
    {"input": "我叫小明，我想学 Java"},
    config={"configurable": {"session_id": "session_abc"}},
)
print(f"AI: {response1}")

# 第二轮 - AI 应该记得"小明"和"Java"
response2 = chain_with_history.invoke(
    {"input": "你还记得我的名字吗？我想学什么语言？"},
    config={"configurable": {"session_id": "session_abc"}},
)
print(f"AI: {response2}")
```

输出：

```
AI: 你好，小明！欢迎学习Java。建议从基础语法开始，比如变量、循环、面向对象。推荐《Head First Java》或菜鸟教程。有什么具体问题随时问我！
AI: 当然记得！你叫小明，想学Java语言。需要学习建议或入门资料，随时告诉我！
```

`session_id` 是区分不同用户/不同会话的关键。换个 session_id，AI 就不记得之前的对话了：

```python
# 新会话 - AI 不知道你是谁
response3 = chain_with_history.invoke(
    {"input": "你知道我是谁吗？"},
    config={"configurable": {"session_id": "session_xyz"}},
)
print(f"AI: {response3}")
# 输出: AI: 抱歉，我无法识别用户的身份。每次对话都是独立的，我没有记忆。请告诉我你需要什么帮助！
```

`RunnableWithMessageHistory` 的几个关键参数：

| 参数 | 说明 |
|------|------|
| `runnable` | 要包装的 Chain（必须是 Runnable） |
| `get_session_history` | 函数，接收 session_id，返回 ChatMessageHistory 对象 |
| `input_messages_key` | 用户输入在 invoke 参数中的 key 名，比如 "input" |
| `history_messages_key` | 历史消息在 prompt 中的占位符 key 名，比如 "history" |

`input_messages_key` 和 `history_messages_key` 告诉 LangChain："从 invoke 的参数里取 `input` 字段作为用户消息，把历史消息塞到 prompt 的 `history` 位置"。

这里很容易有一个疑问：既然 `history_messages_key="history"` 已经告诉 LangChain 把历史消息放到哪里了，为什么还要指定 `input_messages_key="input"`？

原因是：Chain 的输入不一定只有一句用户消息。

比如你的 Chain 可能这样调用：

```python
chain.invoke({
    "input": "我叫小明",
    "language": "Java",
    "style": "适合新手",
})
```

这里真正应该写入聊天历史的是 `"input": "我叫小明"`，而不是 `language`、`style` 这些业务参数。LangChain 不能靠字段名猜测哪一个是用户本轮说的话，所以需要你明确指定：

```python
input_messages_key="input"
```

再结合前面的 prompt 看：

```python
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个友好的编程助手，用中文回答问题。回答要简洁。"),
    MessagesPlaceholder("history"),
    ("human", "{input}"),
])
```

`RunnableWithMessageHistory` 每次调用前后大致做的是这几步：

```python
# 调用前：取出当前 session 的历史消息
history = get_session_history(session_id)

# 调用前：把历史消息塞进 prompt 的 history 占位符
chain_input = {
    "input": "我叫小明",
    "history": history.messages,
}

# 调用后：把本轮用户输入和 AI 回复写回历史
history.add_user_message("我叫小明")
history.add_ai_message(response)
```

所以两个参数的方向不同：

| 参数 | 解决的问题 |
|------|------------|
| `history_messages_key` | 已有历史消息要放到 prompt 的哪个位置 |
| `input_messages_key` | 本轮 invoke 参数里，哪一个字段要作为用户消息写入历史 |

如果不指定 `input_messages_key`，遇到这种输入就会有歧义：

```python
chain.invoke({
    "question": "你还记得我吗？",
    "context": "用户正在学习 Java",
})
```

到底应该把 `question` 写入历史，还是把 `context` 也写进去？如果自动猜，很容易把检索上下文、配置参数、临时变量都错误写入聊天历史。

简单记：`history_messages_key` 负责"把旧消息放进 Prompt"，`input_messages_key` 负责"把本轮新消息存进历史"。

## trim_messages：对话太长了怎么办

多轮对话不断累积，历史消息会越来越长。全部发给 LLM 有两个问题：一是超过模型的上下文窗口会报错，二是 token 消耗越来越大，费用蹭蹭涨。

LangChain 提供了 `trim_messages` 函数来裁剪消息列表：

```python
from langchain_core.messages import trim_messages, SystemMessage, HumanMessage, AIMessage

messages = [
    SystemMessage(content="你是一个编程助手"),
    HumanMessage(content="什么是变量？"),
    AIMessage(content="变量是存储数据的容器。"),
    HumanMessage(content="什么是函数？"),
    AIMessage(content="函数是可复用的代码块。"),
    HumanMessage(content="什么是类？"),
    AIMessage(content="类是面向对象编程的基础。"),
    HumanMessage(content="什么是模块？"),
    AIMessage(content="模块是组织代码的方式。"),
    HumanMessage(content="什么是装饰器？"),
    AIMessage(content="装饰器是修改函数行为的语法糖。"),
]

# 裁剪：只保留最近 5 条消息，但 system 消息始终保留
trimmed = trim_messages(
    messages,
    max_tokens=5,        # 最多保留 5 条（这里用 len 做计数）
    strategy="last",     # 保留最近的
    token_counter=len,   # 用消息条数计数
    include_system=True, # 始终保留 system 消息
    allow_partial=False, # 不截断单条消息
)
```

输出：

```
[SystemMessage] 你是一个编程助手
[HumanMessage] 什么是模块？
[AIMessage] 模块是组织代码的方式。
[HumanMessage] 什么是装饰器？
[AIMessage] 装饰器是修改函数行为的语法糖。
```

原始 11 条消息被裁剪到了 5 条。注意 system 消息始终保留（因为 `include_system=True`），所以实际被保留的对话轮次是 2 轮（4 条对话消息 + 1 条 system 消息 = 5 条）。

`trim_messages` 的参数说明：

| 参数 | 说明 |
|------|------|
| `messages` | 要裁剪的消息列表 |
| `max_tokens` | 保留的最大 token 数（或消息条数，取决于 token_counter） |
| `strategy` | `"last"` 保留最近的消息，`"first"` 保留最早的消息 |
| `token_counter` | 计数方式。`len` 按消息条数计数；传 LLM 对象则按真实 token 数计数 |
| `include_system` | 是否始终保留 system 消息，不受 max_tokens 限制 |
| `allow_partial` | 是否允许截断单条消息（通常设为 False） |

这里重点看一下 `token_counter`。它决定 `max_tokens` 到底按什么计数。

本文示例里写的是：

```python
token_counter=len
```

这不是按真实 token 数计算，而是按消息条数计算。因为 `len(messages)` 返回的是消息列表长度，所以 `max_tokens=5` 的意思其实是"最多保留 5 条消息"。

生产环境更常见的是传 LLM 对象：

```python
token_counter=llm
```

这时 LangChain 会调用模型对象的 token 计数能力，按真实 token 数估算上下文长度。这样 `max_tokens=2000` 才是真正意义上的"最多保留 2000 个 token"。

还有一种写法是：

```python
token_counter="approximate"
```

`approximate` 表示近似 token 计数。它不会调用具体模型的精确 token 计算逻辑，而是用 LangChain 内置的快速估算方式来判断消息大概占多少 token。它的优点是快，适合每次请求前都要裁剪历史的高频路径；缺点是不如模型自己的 token 计数精确。

简单对比：

| token_counter | 计数方式 | 适合场景 |
|---------------|----------|----------|
| `len` | 按消息条数计数 | 教学示例、快速演示 |
| `llm` | 按模型的真实 token 计数 | 生产环境，需要控制上下文长度 |
| `"approximate"` | 近似 token 计数 | 高频调用，想减少精确计数开销 |

`allow_partial` 控制的是"一条消息本身能不能被切开"。

比如有一条很长的用户消息：

```python
HumanMessage(content="请帮我分析下面这段很长的代码......")
```

如果 `allow_partial=False`，这条消息要么完整保留，要么完整丢弃，不会只保留前半句或后半句。对聊天历史来说，这通常更合适，因为半条用户问题、半条 AI 回复很容易破坏上下文。

如果 `allow_partial=True`，LangChain 可以截断单条消息，只保留其中一部分内容。它适合长文档、长日志、长文章片段这类场景：即使只保留一部分文本，也比完全丢掉更有价值。

不过要注意：`allow_partial` 有没有实际意义，取决于 `token_counter` 的计数方式。

前面示例里用的是 `token_counter=len`，也就是按消息条数计数。此时每条消息都算 1：

```text
1 条 HumanMessage = 1
1 条 AIMessage = 1
```

既然裁剪单位是整条消息，`allow_partial=True` 基本没有实际价值。它不会突然把"一条消息"按 token 切成半条来算，因为 `len` 根本不看消息内容有多长。

`allow_partial` 真正有用，是在按 token 数或内容长度裁剪时：

```python
trimmed = trim_messages(
    messages,
    max_tokens=2000,
    strategy="last",
    token_counter=llm,        # 按真实 token 数计数
    allow_partial=True,
)
```

比如最后只剩 300 个 token 空间，但某条消息有 1000 个 token：

| 参数 | 结果 |
|------|------|
| `allow_partial=False` | 这条消息放不下，整条丢弃 |
| `allow_partial=True` | 截取这条消息的一部分，尽量塞进剩余 token 空间 |

不一定非得传 LLM。只要 `token_counter` 是按内容长度计数，`allow_partial` 就有意义。比如传 `"approximate"` 做近似 token 计数，或者传一个自定义字符计数函数，都可以让 `allow_partial=True` 发挥作用。

简单判断：

| token_counter | allow_partial 是否有实际意义 | 原因 |
|---------------|-----------------------------|------|
| `len` | 基本没有 | 按消息条数裁剪，不看单条消息长度 |
| `llm` | 有 | 按真实 token 数裁剪 |
| `"approximate"` | 有 | 近似按 token 数裁剪 |
| 自定义内容长度计数函数 | 有 | 可以判断单条消息是否超出剩余空间 |

实际使用建议：

| 场景 | 推荐值 | 原因 |
|------|--------|------|
| 普通聊天记忆 | `False` | 保持每条对话完整 |
| 代码问答 | `False` | 半段代码容易造成误解 |
| 长文档摘要 | `True` | 截取一部分内容仍然有用 |
| 日志分析 | `True` | 可以保留最近或最关键的一段 |

示例中 `token_counter=len` 表示按消息条数计数，每条消息算 1。生产环境中可以传 LLM 对象作为 counter，LangChain 会自动计算真实的 token 数：

```python
# 生产环境的用法：用 LLM 计算真实 token 数
trimmed = trim_messages(
    messages,
    max_tokens=2000,
    strategy="last",
    token_counter=llm,        # 传 LLM 对象，自动计算真实 token
    include_system=True,
    allow_partial=False,      # 聊天历史通常保持单条消息完整
)
```

如果处理的是长文档或日志片段，而不是普通聊天历史，可以把 `allow_partial` 改成 `True`，让 LangChain 尽量利用剩余 token 空间。

## Chain + History + trim：完整组合

把前面三个知识点串起来：Chain 执行对话，RunnableWithMessageHistory 管理历史，trim_messages 控制长度。

```python
llm = ChatDeepSeek(
    model="deepseek-v4-flash",
    api_key=DEEPSEEK_API_KEY,
    base_url=DEEPSEEK_BASE_URL,
)

store = {}

def get_trimmed_history(session_id: str) -> ChatMessageHistory:
    """获取历史记录，超过 4 条就裁剪"""
    if session_id not in store:
        store[session_id] = ChatMessageHistory()

    raw_history = store[session_id]
    if len(raw_history.messages) > 4:
        trimmed = trim_messages(
            raw_history.messages,
            max_tokens=4,
            strategy="last",
            token_counter=len,
            include_system=False,
            allow_partial=False,
        )
        new_history = ChatMessageHistory()
        for msg in trimmed:
            new_history.add_message(msg)
        return new_history

    return raw_history

prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个编程助手，用中文简洁回答。"),
    MessagesPlaceholder("history"),
    ("human", "{input}"),
])

chain = prompt | llm | StrOutputParser()

chain_with_history = RunnableWithMessageHistory(
    chain,
    get_trimmed_history,
    input_messages_key="input",
    history_messages_key="history",
)
```

关键在 `get_trimmed_history` 函数：它先从 store 取出原始历史，如果超过 4 条就裁剪，返回裁剪后的版本。注意这里 `include_system=False`，因为历史消息里没有 system 消息（system 消息在 prompt 模板里，不在历史记录中）。

连续发 6 轮对话测试一下：

```python
questions = [
    "什么是 Python 的列表？",
    "什么是字典？",
    "什么是元组？",
    "什么是集合？",
    "什么是生成器？",
    "回顾一下：我前面问过列表和字典，你还记得吗？",
]

for i, question in enumerate(questions, 1):
    response = chain_with_history.invoke(
        {"input": question},
        config={"configurable": {"session_id": "trim_demo"}},
    )
    print(f"第{i}轮 - 用户: {question}")
    print(f"第{i}轮 - AI: {response}")
    print(f"(历史消息数: {len(store['trim_demo'].messages)})\n")
```

输出（节选）：

```
第1轮 - 用户: 什么是 Python 的列表？
第1轮 - AI: Python 的列表是一种有序、可变的数据结构，用方括号 `[]` 表示，可以存储任意类型的元素...
(历史消息数: 2)

第2轮 - 用户: 什么是字典？
第2轮 - AI: Python 的字典是一种无序、可变的数据结构，用花括号 `{}` 表示，以键值对（key: value）形式存储数据...
(历史消息数: 4)

第3轮 - 用户: 什么是元组？
第3轮 - AI: Python 的元组是一种有序、不可变的数据结构...
(历史消息数: 6)  <- 超过 4 条，触发裁剪

第6轮 - 用户: 回顾一下：我前面问过列表和字典，你还记得吗？
第6轮 - AI: 你之前问过字典和元组，但还没有问过列表哦...
(历史消息数: 6)  <- 裁剪后最早的对话被丢弃，AI 记忆不完整
```

这里有个重要区分：`store` 里存的是完整历史（12 条），但 `get_trimmed_history` 每次返回给 Chain 的只有最近 4 条。完整历史一直在增长，但发给 LLM 的消息始终受控。

## 对比：手写滑动窗口 vs LangChain 的做法

在 AI Agent 教程中，我们手写过滑动窗口裁剪：

```python
# AI Agent 教程中的手写裁剪
def trim_messages(messages: list, max_rounds: int) -> list:
    system_messages = [m for m in messages if m["role"] == "system"]
    conversation = [m for m in messages if m["role"] != "system"]
    rounds = []
    current_round = []
    for msg in conversation:
        if msg["role"] == "user" and current_round:
            rounds.append(current_round)
            current_round = []
        current_round.append(msg)
    if current_round:
        rounds.append(current_round)
    trimmed = rounds[-max_rounds:]
    flat = [msg for round_msgs in trimmed for msg in round_msgs]
    return system_messages + flat
```

对比两种方式：

- 手写方式需要自己处理按轮次分组、system 消息特殊处理、边界情况等细节，大约 15 行代码
- LangChain 的 `trim_messages` 一行调用就搞定，支持按 token 数计数、多种策略、system 消息自动保留
- `RunnableWithMessageHistory` 进一步把"存历史 + 取历史 + 注入 Chain"整个流程封装好了，不用手动管理消息列表

本质上做的是同一件事——保留最近的消息、丢掉早期的、始终保留 system 消息。区别在于 LangChain 提供了开箱即用的工具，而不用自己写裁剪逻辑。

## 总结

这节学了三个组件：

- `ChatMessageHistory`：存消息的容器，支持增删查清
- `RunnableWithMessageHistory`：包装 Chain，自动管理历史消息的读取和写入，通过 session_id 区分不同会话
- `trim_messages`：按 token 数或消息条数裁剪历史，支持多种策略，防止上下文过长

三个组件各司其职：ChatMessageHistory 负责"存"，RunnableWithMessageHistory 负责"接入"，trim_messages 负责"控制长度"。
