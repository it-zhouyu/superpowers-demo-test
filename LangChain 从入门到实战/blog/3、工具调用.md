前两篇我们搭出了完整的 LCEL 链，能动态拼提示词、能解析结构化输出、能并行执行。但有一个根本限制：模型只能"说话"，不能"做事"。你问它"北京天气怎么样"，它要么编一个，要么说"我无法获取实时数据"。

工具调用（Tool Calling）就是解决这个问题的。它的思路很简单：模型不直接回答，而是告诉你要调用哪个函数、传什么参数；你执行完函数，把结果喂回模型，模型再基于结果生成最终回复。

本文从零实现完整的工具调用流程，包括定义工具、绑定工具、手动执行工具循环、结构化输出。

## 用 @tool 定义工具函数

LangChain 用 `@tool` 装饰器把普通 Python 函数变成"工具"。来看两个例子：

```python
import json
from langchain_core.tools import tool

@tool
def get_weather(city: str) -> str:
    """查询指定城市的天气信息。"""
    mock_data = {
        "北京": "晴天，气温 18°C，空气质量良好",
        "上海": "多云，气温 22°C，有轻微雾霾",
        "深圳": "阴天，气温 26°C，可能有小雨",
    }
    weather = mock_data.get(city, f"暂无{city}的天气数据")
    return json.dumps({"city": city, "weather": weather}, ensure_ascii=False)

@tool
def get_time(city: str, timezone_str: str = "Asia/Shanghai") -> str:
    """查询指定城市的当前时间。"""
    tz_map = {
        "Asia/Shanghai": timedelta(hours=8),
        "America/New_York": timedelta(hours=-4),
    }
    offset = tz_map.get(timezone_str, timedelta(hours=8))
    tz = timezone(offset)
    now = datetime.now(tz)
    return json.dumps({
        "city": city,
        "timezone": timezone_str,
        "current_time": now.strftime("%Y-%m-%d %H:%M:%S"),
    }, ensure_ascii=False)
```

这两个函数就是普通的 Python 函数，唯一的区别是加上了 `@tool` 装饰器。但这个装饰器做了两件重要的事情：

1. **提取函数签名**：参数名、类型注解、默认值
2. **提取函数文档**：`"""..."""` 中的 docstring 作为工具描述

LangChain 把这些信息自动生成一份 JSON Schema，告诉模型"这个工具叫什么名字、接收什么参数、每个参数是什么类型"。模型根据这份 Schema 和用户的问题，决定是否调用工具、调用哪个、传什么参数。

**这里有一个关键点**：函数的类型注解不是可选的装饰，而是工具调用的核心组成部分。对比一下：

```python
# 正确写法：有类型注解，LangChain 能生成精确的 Schema
@tool
def get_weather(city: str) -> str:
    """查询指定城市的天气信息。"""

# 错误写法：没有类型注解，LangChain 不知道参数类型
@tool
def get_weather(city) -> str:  # 缺少 str 类型注解
    """查询指定城市的天气信息。"""
```

`@tool` 装饰器的关键要素：

| 要素 | 作用 | 是否必须 |
|------|------|----------|
| 函数名 | 作为工具的名称，模型用它来选择调用哪个工具 | 是 |
| docstring | 作为工具的描述，模型用它理解工具的用途 | 是 |
| 参数类型注解 | 生成 JSON Schema 中的参数类型 | 是 |
| 参数默认值 | 参数可选，模型不传时使用默认值 | 否 |
| `Field(description=...)` | 给参数加描述说明 | 否 |

## bind_tools：把工具绑定到模型

定义好工具后，需要告诉模型"你有这些工具可以用"。`bind_tools()` 做这件事：

```python
llm = ChatDeepSeek(
    model="deepseek-v4-flash",
    api_key="sk-xxx",
    base_url="https://api.deepseek.com",
    temperature=0,
)

llm_with_tools = llm.bind_tools([get_weather, get_time])
```

`bind_tools` 不会立即调用任何东西。它的作用是把工具的 Schema 信息附加到模型客户端上，之后每次调用模型时，这些 Schema 会自动放在请求里发给模型。

这里还可以传一个重要参数：`tool_choice`。

`tools` 负责告诉模型"有哪些工具"，`tool_choice` 负责告诉模型"这次该怎么使用这些工具"。它控制的是模型输出阶段的选择权，不是 Python 代码里的函数执行逻辑。

比如默认情况下，我们通常不写 `tool_choice`：

```python
llm_with_tools = llm.bind_tools([get_weather, get_time])
```

这时等价于让模型自己判断：如果用户问题需要工具，就返回 `tool_calls`；如果不需要工具，就直接返回文本。比如问"北京天气怎么样"，模型可能调用 `get_weather`；问"你好"，模型通常直接回复文本。

如果你希望模型必须调用工具，可以这样写：

```python
llm_with_tools = llm.bind_tools(
    [get_weather, get_time],
    tool_choice="required",
)
```

这表示本轮回复必须包含至少一个工具调用。适合这类场景：用户已经进入一个明确需要外部数据的流程，比如"查询订单状态"、"读取数据库记录"、"调用搜索接口"。

如果你希望模型只能调用某一个指定工具，可以这样写：

```python
llm_with_tools = llm.bind_tools(
    [get_weather, get_time],
    tool_choice="get_weather",
)
```

这表示模型不能自由选择 `get_time`，只能生成 `get_weather` 的调用。`with_structured_output(method="function_calling")` 底层就是利用这个思路：把输出结构伪装成一个工具，再用 `tool_choice` 强制模型"调用"这个工具。

常见取值可以这样理解：

| tool_choice | 作用 | 适合场景 |
|-------------|------|----------|
| 不传或 `"auto"` | 模型自己决定是否调用工具 | 普通 Agent 对话 |
| `"none"` | 禁止模型调用工具，只能输出文本 | 临时关闭工具能力 |
| `"required"` | 模型必须调用至少一个工具 | 本轮必须查外部数据 |
| 指定工具名 | 模型必须调用这个工具 | 强制结构化输出、固定流程 |

上面是 LangChain 的写法。如果直接调用 OpenAI / DeepSeek 兼容 API，指定某个工具时通常写成这样的对象：

```python
tool_choice = {
    "type": "function",
    "function": {"name": "get_weather"},
}
```

注意：`tool_choice` 只是约束模型返回什么样的消息。模型返回 `tool_calls` 以后，工具函数仍然不会自动执行，还是需要你写后面的工具执行循环。

现在试一下：

```python
from langchain_core.messages import HumanMessage

messages = [HumanMessage(content="北京今天天气怎么样？")]
ai_message = llm_with_tools.invoke(messages)

print(f"content: {ai_message.content}")
print(f"tool_calls: {ai_message.tool_calls}")
```

输出：

```
content:
tool_calls: [{'name': 'get_weather', 'args': {'city': '北京'}, 'id': 'chatcmpl-tool-7a1b2c3d'}]
```

注意两点：

1. `content` 是空的——模型没有直接回复文本，而是选择了调用工具
2. `tool_calls` 是一个列表，包含模型想要调用的工具信息

`tool_calls` 列表中每个元素的结构：

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | str | 模型选择调用的工具名称 |
| `args` | dict | 模型生成的调用参数 |
| `id` | str | 本次工具调用的唯一标识，后续回复时需要对应 |

模型没有执行任何函数，它只是说"我想调用 `get_weather`，参数是 `{'city': '北京'}`"。真正的执行需要你来做。

## 手动工具执行循环

理解了单个工具调用的流程，下面实现完整的循环：模型选择工具 -> 执行 -> 把结果喂回 -> 模型生成最终回复。

```python
from langchain_core.messages import ToolMessage

def run_with_tools(llm_with_tools, messages: list) -> str:
    # 建立工具名到函数的映射
    tools_map = {t.name: t for t in [get_weather, get_time]}

    while True:
        # 1. 调用模型
        ai_message = llm_with_tools.invoke(messages)
        messages.append(ai_message)

        # 2. 如果没有工具调用，返回最终文本回复
        if not ai_message.tool_calls:
            return ai_message.content

        # 3. 逐个执行工具
        for tool_call in ai_message.tool_calls:
            tool_name = tool_call["name"]
            tool_args = tool_call["args"]

            print(f"  [调用工具] {tool_name}({tool_args})")

            # 执行工具函数
            tool_fn = tools_map[tool_name]
            tool_result = tool_fn.invoke(tool_args)

            print(f"  [工具返回] {tool_result}")

            # 4. 创建 ToolMessage，id 必须对应
            tool_message = ToolMessage(
                content=tool_result,
                tool_call_id=tool_call["id"],
            )
            messages.append(tool_message)
        # 5. 回到第 1 步，继续调用模型
```

整个循环用 `while True` 驱动，每次迭代做五件事：

1. 把当前消息列表发给模型
2. 检查模型回复中有没有 `tool_calls`
3. 如果有，逐个执行对应的工具函数
4. 把工具结果包装成 `ToolMessage` 追加到消息列表
5. 回到第 1 步，模型看到工具结果后再决定：继续调工具还是给出最终回复

注意 `ToolMessage` 的 `tool_call_id` 必须和 `ai_message.tool_calls` 中的 `id` 一致，这样模型才能把工具结果和对应的调用匹配起来。

用一下这个函数：

```python
messages = [HumanMessage(content="北京现在几点了？天气怎么样？")]
final_answer = run_with_tools(llm_with_tools, messages)
print(f"最终回复: {final_answer}")
```

执行过程：

```
  [调用工具] get_time({'city': '北京'})
  [工具返回] {"city": "北京", "timezone": "Asia/Shanghai", "current_time": "2026-06-04 14:30:00"}
  [调用工具] get_weather({'city': '北京'})
  [工具返回] {"city": "北京", "weather": "晴天，气温 18°C，空气质量良好"}
最终回复: 北京现在是2026年6月4日14:30（Asia/Shanghai时区），天气为晴天，气温18°C，空气质量良好。
```

用户问了两个问题（时间 + 天气），模型一次性调用了两个工具，然后把结果综合成一段自然语言回复。这正是工具调用的价值：模型自己决定需要哪些信息，你只需要提供工具。

## 对比：LangChain @tool vs 手写 JSON Schema

如果你之前看过 AI Agent 教程中的示例代码，可能会记得手写 JSON Schema 的方式。对比一下两种写法。

**手写 JSON Schema（原生 API 风格）**：

```python
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "查询指定城市的天气信息",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "城市名称",
                }
            },
            "required": ["city"],
        },
    },
}]
```

**LangChain @tool 风格**：

```python
@tool
def get_weather(city: str) -> str:
    """查询指定城市的天气信息。"""
```

两者做的事情完全一样：告诉模型"有这个工具可以用"。区别在于：

| 对比维度 | 手写 JSON Schema | LangChain @tool |
|----------|------------------|-----------------|
| 代码量 | 多，每个工具都要写一大段 JSON | 少，一个装饰器 + 函数定义 |
| 维护成本 | 函数签名改了要同步改 JSON | 改函数签名就行，Schema 自动更新 |
| 类型安全 | 手写字符串，写错了不会报错 | Python 类型注解，IDE 能检查 |
| 灵活性 | 完全可控，可以加各种 JSON Schema 特性 | 覆盖常见场景，极少数复杂 Schema 需要额外配置 |

如果你的项目已经用了 LangChain，`@tool` 是更好的选择。如果用的是原生 OpenAI SDK 或者需要精确控制 Schema 的每一个字段，手写 JSON Schema 更合适。

## with_structured_output：强制结构化输出

工具调用还有一个常用场景：不是真的要执行什么函数，而是想强制模型返回结构化数据。

比如让模型推荐一本书，你希望输出是一个固定格式的对象而不是一段自由文本。`with_structured_output()` 就是干这个的：

```python
from pydantic import BaseModel, Field

class BookRecommendation(BaseModel):
    title: str = Field(description="书名")
    author: str = Field(description="作者")
    genre: str = Field(description="类型/分类")
    reason: str = Field(description="推荐理由，一句话")

# DeepSeek V4 Flash 的思考模式与 tool_choice 不兼容，需要用 json_mode
structured_llm = llm.with_structured_output(BookRecommendation, method="json_mode")

# DeepSeek 的 json_mode 要求 prompt 中包含 "json" 关键词
result = structured_llm.invoke(
    "请以 json 格式推荐一本适合程序员看的科幻小说，"
    "包含 title、author、genre、reason 四个字段"
)
print(result.title)    # 三体
print(result.author)   # 刘慈欣
print(result.genre)    # 科幻
print(result.reason)   # 硬科幻小说...
```

不需要自己定义提示词模板，不需要自己写输出解析器，一行代码就搞定了。但这不是模型突然"懂了 Pydantic"，而是 LangChain 在中间自动做了一层封装。

先看 `BookRecommendation` 这个例子。我们传进去的是一个 Pydantic 模型：

```python
class BookRecommendation(BaseModel):
    title: str = Field(description="书名")
    author: str = Field(description="作者")
    genre: str = Field(description="类型/分类")
    reason: str = Field(description="推荐理由，一句话")
```

LangChain 拿到这个类以后，会先把它转换成模型 API 能理解的结构。大致等价于下面这个 Schema：

```python
{
    "type": "function",
    "function": {
        "name": "BookRecommendation",
        "description": "",
        "parameters": {
            "type": "object",
            "properties": {
                "title": {
                    "type": "string",
                    "description": "书名",
                },
                "author": {
                    "type": "string",
                    "description": "作者",
                },
                "genre": {
                    "type": "string",
                    "description": "类型/分类",
                },
                "reason": {
                    "type": "string",
                    "description": "推荐理由，一句话",
                },
            },
            "required": ["title", "author", "genre", "reason"],
        },
    },
}
```

这里的映射关系很直接：

| Pydantic 写法 | 转成工具 Schema 后 |
|--------------|-------------------|
| 类名 `BookRecommendation` | 工具名 `BookRecommendation` |
| 字段名 `title` | 参数名 `title` |
| 类型 `str` | JSON Schema 类型 `string` |
| `Field(description="书名")` | 参数说明 `description: "书名"` |
| 没有默认值的字段 | 放进 `required` 必填字段 |

如果使用默认的 `method="function_calling"`，LangChain 底层会继续做 3 件事：

1. 调用 `bind_tools([BookRecommendation], tool_choice="BookRecommendation")`，把这个 Pydantic 模型绑定成一个工具，并要求模型必须调用这个工具。
2. 模型返回的原始消息不是普通文本，而是一个 `AIMessage`，里面带着 `tool_calls`，工具参数就是模型生成的结构化数据。
3. LangChain 再用 `PydanticToolsParser` 读取 `tool_calls[0]["args"]`，把参数校验并转换成 `BookRecommendation` 实例。

所以默认路径可以理解成：

```python
BookRecommendation
    -> OpenAI Tool Schema
    -> bind_tools(..., tool_choice="BookRecommendation")
    -> AIMessage(tool_calls=[...])
    -> PydanticToolsParser
    -> BookRecommendation(...)
```

这就是为什么返回值可以直接写 `result.title`。模型没有真的执行一个叫 `BookRecommendation` 的 Python 函数，它只是按这个"工具参数格式"填了一份 JSON，LangChain 再把 JSON 解析成 Pydantic 对象。

不过本文示例里写的是：

```python
structured_llm = llm.with_structured_output(BookRecommendation, method="json_mode")
```

这条路径和默认工具调用不一样。`json_mode` 不会把 `BookRecommendation` 作为工具绑定给模型，而是给模型请求参数加上：

```python
response_format = {"type": "json_object"}
```

然后 LangChain 使用 `PydanticOutputParser` 解析模型返回的 JSON 文本，校验字段是否符合 `BookRecommendation`。所以 `json_mode` 的流程是：

```python
BookRecommendation
    -> response_format={"type": "json_object"}
    -> 模型返回 JSON 字符串
    -> PydanticOutputParser
    -> BookRecommendation(...)
```

两条路径的区别在于：`function_calling` 是让模型"调用一个只有参数、没有真实执行逻辑的工具"；`json_mode` 是让模型"直接输出 JSON 文本"，再由 LangChain 解析。

**注意**：DeepSeek V4 Flash 有一个思考模式（thinking mode），与默认的 `tool_choice` 参数不兼容。所以这里需要显式指定 `method="json_mode"`，让模型通过 JSON 模式输出结构化数据。同时，DeepSeek 的 JSON 模式要求 prompt 中包含 "json" 关键词，否则会报错。

返回值直接是 `BookRecommendation` 的实例（Pydantic 对象），不是字典也不是 JSON 字符串，可以直接用 `.title`、`.author` 访问字段。

`with_structured_output()` 常见有 3 种模式：

| method | 底层约束方式 | 解析器 | 适合场景 |
|--------|--------------|--------|----------|
| `function_calling` | 把 Schema 当成工具绑定，并用 `tool_choice` 要求模型调用 | `PydanticToolsParser` 或 `JsonOutputKeyToolsParser` | 模型支持工具调用，并且 `tool_choice` 可用 |
| `json_mode` | 传 `response_format={"type": "json_object"}`，要求模型输出 JSON | `PydanticOutputParser` 或 `JsonOutputParser` | 模型支持 JSON 模式，但工具调用限制较多 |
| `json_schema` | 传更严格的结构化输出 Schema | Pydantic 或 JSON 解析器 | 模型支持原生结构化输出能力 |

在 DeepSeek 这里还有一个细节：`method="json_schema"` 会退回到 `function_calling` 路径；如果设置 `strict=True`，LangChain 会切到 DeepSeek 的 beta API 地址，以便使用更严格的 Schema 校验。

## 工具调用 vs 结构化输出

`bind_tools()` 和 `with_structured_output(method="function_calling")` 底层都是 Tool Calling（工具调用）。如果 `with_structured_output()` 指定 `json_mode` 或 `json_schema`，底层会走模型提供的 JSON / 结构化输出能力，但用途仍然一样：约束模型返回固定格式。

| 维度 | 工具调用（bind_tools） | 结构化输出（with_structured_output） |
|------|----------------------|-------------------------------------|
| 目的 | 让模型调用外部函数 | 让模型返回固定格式的数据 |
| 是否真的执行函数 | 是，需要你写执行逻辑 | 否，只是借用 Schema 约束输出格式 |
| 返回类型 | AIMessage（含 tool_calls） | Pydantic 模型实例 |
| 适用场景 | 查天气、查数据库、调 API | 信息抽取、分类、生成表单数据 |
| 灵活性 | 可以绑定多个工具，模型自行选择 | 通常只有一个输出结构，模型必须按它返回 |

简单判断标准：如果你需要模型触发一个真实的动作（调用 API、查数据库），用 `bind_tools`；如果你只是想要固定格式的输出，用 `with_structured_output`。

## 小结

- **@tool 装饰器**把普通 Python 函数变成工具，自动从函数签名和 docstring 生成 JSON Schema
- **bind_tools()** 把工具的 Schema 绑定到模型，模型根据用户问题决定是否调用工具
- **tool_choice** 控制模型是否可以调用工具、是否必须调用工具、是否必须调用指定工具
- **手动工具循环**：解析 `tool_calls` -> 执行函数 -> 构造 `ToolMessage` -> 循环，直到模型给出最终回复
- **with_structured_output()** 把 Pydantic 模型转换成模型 API 可理解的结构，再通过工具调用、JSON 模式或结构化输出模式约束模型返回固定格式
- 相比手写 JSON Schema，`@tool` 用类型注解自动生成 Schema，代码更简洁、更容易维护

## 完整代码

把下面代码保存为 `step3_tool_calling.py`，并和前面的 `config.py` 放在同一个目录下运行：

```python
"""
Step 3: Tool Calling — 工具调用

本文件演示：
- 用 @tool 装饰器定义工具函数
- llm.bind_tools() 把工具绑定到模型
- 手动执行工具调用循环（解析 tool_calls -> 执行 -> ToolMessage -> 循环）
- with_structured_output() 获取结构化输出
"""

import json
from datetime import datetime, timezone, timedelta
from typing import Optional

from pydantic import BaseModel, Field
from langchain_deepseek import ChatDeepSeek
from langchain_core.tools import tool
from langchain_core.messages import HumanMessage, ToolMessage

from config import DEEPSEEK_API_KEY, DEEPSEEK_MODEL, DEEPSEEK_BASE_URL

# ============================================================
# 定义工具函数
# ============================================================

@tool
def get_weather(city: str) -> str:
    """查询指定城市的天气信息。"""
    # 这里用模拟数据演示，实际项目中会调用天气 API
    mock_data = {
        "北京": "晴天，气温 18°C，空气质量良好",
        "上海": "多云，气温 22°C，有轻微雾霾",
        "深圳": "阴天，气温 26°C，可能有小雨",
    }
    weather = mock_data.get(city, f"暂无{city}的天气数据")
    return json.dumps({"city": city, "weather": weather}, ensure_ascii=False)

@tool
def get_time(city: str, timezone_str: str = "Asia/Shanghai") -> str:
    """查询指定城市的当前时间。"""
    # 根据 timezone_str 计算对应时区的当前时间
    tz_map = {
        "Asia/Shanghai": timedelta(hours=8),
        "America/New_York": timedelta(hours=-4),
        "Europe/London": timedelta(hours=1),
        "Asia/Tokyo": timedelta(hours=9),
    }
    offset = tz_map.get(timezone_str, timedelta(hours=8))
    tz = timezone(offset)
    now = datetime.now(tz)
    return json.dumps({
        "city": city,
        "timezone": timezone_str,
        "current_time": now.strftime("%Y-%m-%d %H:%M:%S"),
    }, ensure_ascii=False)

def run_with_tools(llm_with_tools, messages: list) -> str:
    """
    执行完整的工具调用循环：
    1. 把消息发给模型
    2. 如果模型返回 tool_calls，逐个执行工具
    3. 把工具结果作为 ToolMessage 追加到消息列表
    4. 重复，直到模型不再调用工具，返回最终文本回复
    """
    tools_map = {t.name: t for t in [get_weather, get_time]}

    while True:
        # 调用模型
        ai_message = llm_with_tools.invoke(messages)
        messages.append(ai_message)

        # 检查是否有工具调用
        if not ai_message.tool_calls:
            return ai_message.content

        # 逐个执行工具
        for tool_call in ai_message.tool_calls:
            tool_name = tool_call["name"]
            tool_args = tool_call["args"]

            print(f"  [调用工具] {tool_name}({tool_args})")

            # 执行工具函数
            tool_fn = tools_map[tool_name]
            tool_result = tool_fn.invoke(tool_args)

            print(f"  [工具返回] {tool_result}")

            # 创建 ToolMessage，tool_call_id 必须和 AIMessage 中的 id 对应
            tool_message = ToolMessage(
                content=tool_result,
                tool_call_id=tool_call["id"],
            )
            messages.append(tool_message)

def main():
    llm = ChatDeepSeek(
        model=DEEPSEEK_MODEL,
        api_key=DEEPSEEK_API_KEY,
        base_url=DEEPSEEK_BASE_URL,
        temperature=0,
    )

    # ============================================================
    # Demo 1: bind_tools — 绑定工具后观察模型响应
    # ============================================================
    print("=" * 50)
    print("Demo 1: llm.bind_tools() 观察工具调用响应")
    print("=" * 50)

    # bind_tools 把工具的函数签名和描述发送给模型
    # 模型会根据用户问题决定是否调用工具
    llm_with_tools = llm.bind_tools([get_weather, get_time])

    messages = [HumanMessage(content="北京今天天气怎么样？")]
    ai_message = llm_with_tools.invoke(messages)

    print(f"\n[模型回复类型] {type(ai_message).__name__}")
    print(f"[回复内容] {ai_message.content}")
    print(f"[工具调用] {ai_message.tool_calls}")

    if ai_message.tool_calls:
        tc = ai_message.tool_calls[0]
        print(f"\n[tool_call 详情]")
        print(f"  name: {tc['name']}")
        print(f"  args: {tc['args']}")
        print(f"  id:   {tc['id']}")

    # ============================================================
    # Demo 2: 手动工具执行循环
    # ============================================================
    print("\n" + "=" * 50)
    print("Demo 2: 手动工具执行循环")
    print("=" * 50)

    messages = [HumanMessage(content="北京现在几点了？天气怎么样？")]
    print(f"\n[用户问题] 北京现在几点了？天气怎么样？")

    final_answer = run_with_tools(llm_with_tools, messages)
    print(f"\n[最终回复] {final_answer}")

    # ============================================================
    # Demo 3: with_structured_output — 结构化输出
    # ============================================================
    print("\n" + "=" * 50)
    print("Demo 3: with_structured_output() 结构化输出")
    print("=" * 50)

    # 定义期望的输出结构
    class BookRecommendation(BaseModel):
        title: str = Field(description="书名")
        author: str = Field(description="作者")
        genre: str = Field(description="类型/分类")
        reason: str = Field(description="推荐理由，一句话")

    # with_structured_output 内部使用 tool calling 机制
    # 模型会被约束为按照 Pydantic 模型的结构输出
    # DeepSeek V4 Flash 的思考模式不支持 tool_choice，需要使用 json_mode
    # 注意: DeepSeek 的 json_mode 要求 prompt 中包含 "json" 关键词
    structured_llm = llm.with_structured_output(BookRecommendation, method="json_mode")

    result = structured_llm.invoke(
        "请以 json 格式推荐一本适合程序员看的科幻小说，"
        "包含 title、author、genre、reason 四个字段"
    )
    print(f"\n[结构化结果] 类型: {type(result).__name__}")
    print(f"  书名: {result.title}")
    print(f"  作者: {result.author}")
    print(f"  类型: {result.genre}")
    print(f"  理由: {result.reason}")

if __name__ == "__main__":
    main()
```
