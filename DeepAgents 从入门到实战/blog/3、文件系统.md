前面，Agent 已经能调用工具了。但工具调用产生的数据放哪？Agent 生成的代码、配置文件写到哪里？Deep Agent 内置了一套文件系统，而且这个文件系统是可插拔的——换一个后端，文件的存储方式就变了。

Deep Agent 提供三种文件系统后端，分别适用于不同场景：

- **StateBackend**：文件存在内存中，最安全，适合对话场景
- **FilesystemBackend**：读写真实磁盘文件，适合本地开发工具
- **SandboxBackend**：在沙箱中隔离执行，适合运行不可信代码

这一篇从最常用的 StateBackend 开始，逐个讲清楚。

## StateBackend：内存文件系统

StateBackend 是 Deep Agent 的默认后端。Agent 通过 `write_file`、`read_file` 等工具操作的文件，全部存储在 LangGraph 的状态（State）中，不会写入真实磁盘。

先看一个完整的例子——创建 Agent，让它写一个文件再读回来确认：

```python
from langchain_deepseek import ChatDeepSeek
from deepagents import create_deep_agent
from deepagents.backends import StateBackend

llm = ChatDeepSeek(
    model="deepseek-chat",
    api_key="your-api-key",
    base_url="https://api.deepseek.com",
)

# StateBackend 是默认后端，这里显式传入是为了演示清晰
agent = create_deep_agent(
    model=llm,
    backend=StateBackend(),
    system_prompt="你是一个文件管理助手。请用中文回复。",
)

task = "请在 notes.txt 中写入以下内容:\n- Deep Agent 是 LangChain 的 Agent 框架\n- 内置文件系统支持\n- 支持子 Agent 和上下文管理\n\n写完后读取文件确认内容。"

result = agent.invoke({"messages": task})
```

执行过程中，Agent 会先调用 `write_file` 把内容写入 `notes.txt`，再调用 `read_file` 读回来确认。我们遍历 `result["messages"]` 可以看到完整过程：

```python
for msg in result["messages"]:
    msg_type = type(msg).__name__
    if msg_type == "AIMessage" and hasattr(msg, "tool_calls") and msg.tool_calls:
        for tc in msg.tool_calls:
            print(f"调用: {tc['name']}({tc['args']})")
    elif msg_type == "ToolMessage":
        print(f"结果: {msg.content[:150]}")
```

输出大致如下：

```
调用: write_file({'file_path': '/notes.txt', 'content': '- Deep Agent 是 LangChain 的 Agent 框架\n- 内置文件系统支持\n- 支持子 Agent 和上下文管理'})
结果: Updated file /notes.txt
调用: read_file({'file_path': '/notes.txt'})
结果:      1	- Deep Agent 是 LangChain 的 Agent 框架
     2	- 内置文件系统支持
     3	- 支持子 Agent 和上下文管理
```

Agent 调用了两个工具：先写后读，确认内容正确。

**StateBackend 的文件存在哪里？** 在 `result["files"]` 字典里：

```python
for path, file_info in result["files"].items():
    if isinstance(file_info, dict):
        content = file_info.get("content", str(file_info))
    else:
        content = str(file_info)
    print(f"{path}:\n{content}")
```

输出：

```
notes.txt:
- Deep Agent 是 LangChain 的 Agent 框架
- 内置文件系统支持
- 支持子 Agent 和上下文管理
```

关键点：文件存在于 Agent 的 LangGraph 状态中，不是真实磁盘上的文件。只要对话线程（Thread）还在，文件就一直在；换个线程或重启程序，文件就没了。

**适用场景**：对话中临时创建文件、代码生成预览、不需要持久化的中间数据。这是最安全的后端，因为 Agent 根本碰不到真实文件系统。

## FilesystemBackend：真实磁盘文件系统

StateBackend 的文件只存在于内存，程序一停就没了。如果需要 Agent 操作真实文件——比如读取项目配置、修改代码文件——就要用 FilesystemBackend。

先创建一个临时目录，放一个示例文件进去：

```python
import os
import tempfile

temp_dir = tempfile.mkdtemp(prefix="deepagents_demo_")

sample_file = os.path.join(temp_dir, "project_info.txt")
with open(sample_file, "w", encoding="utf-8") as f:
    f.write("项目名称: DeepAgents 学习项目\n")
    f.write("版本: 1.0.0\n")
    f.write("作者: 学员\n")
    f.write("描述: 学习 Deep Agent 的文件系统能力\n")
```

然后创建使用 FilesystemBackend 的 Agent：

```python
from deepagents.backends import FilesystemBackend

agent = create_deep_agent(
    model=llm,
    backend=FilesystemBackend(root_dir=temp_dir, virtual_mode=True),
    system_prompt="你是一个文件管理助手，可以读写文件。请用中文回复。",
)
```

这里有两个关键参数：

- `root_dir`：Agent 的根目录，所有文件操作都基于这个目录
- `virtual_mode=True`：启用虚拟路径模式，阻止路径遍历攻击（比如 `../` 跳出根目录、`~` 访问用户目录）

让 Agent 读取刚才创建的文件：

```python
task = "请读取 project_info.txt 文件的内容，然后总结一下这个项目的信息。"
result = agent.invoke({"messages": task})

last_msg = result["messages"][-1]
print(last_msg.content)
```

Agent 会通过 `read_file` 工具读取磁盘上真实的 `project_info.txt`，然后总结内容。这个文件确实存在于 `temp_dir` 中，Agent 读的是真正的磁盘文件。

**两种后端的对比**：

| 特性 | StateBackend | FilesystemBackend |
|------|-------------|-------------------|
| 存储位置 | Agent 状态（内存） | 真实文件系统 |
| 持久性 | 仅在对话线程内 | 程序重启后仍在 |
| 安全性 | 无副作用，不碰真实文件 | 可以修改真实文件 |
| 适用场景 | 对话、预览、临时数据 | 开发工具、CLI 助手 |

## SandboxBackend：沙箱文件系统

前两个后端都在本地执行——StateBackend 在内存里，FilesystemBackend 在本机磁盘上。如果 Agent 需要执行 shell 命令（比如运行 Python 脚本、安装依赖、跑测试），而你又不想让它在你的机器上直接跑，就需要 SandboxBackend。

SandboxBackend 把文件操作和命令执行都放在一个远程沙箱里。Agent 在你的环境中运行，但调用的 `write_file`、`read_file`、`execute` 等工具实际操作的是远程沙箱环境，碰不到你的本机文件。

SandboxBackend 本身不提供沙箱，需要配合第三方沙箱提供商。目前支持四个：

- **Daytona**：开发环境沙箱平台
- **Modal**：Serverless 沙箱平台
- **Runloop**：代码执行沙箱平台
- **LangSmith**：LangChain 自带的沙箱服务

以 Daytona 为例，完整流程如下。

先安装依赖：

```bash
pip install daytona-sdk langchain-daytona
```

创建沙箱并启动 Agent：

```python
from daytona import Daytona
from deepagents import create_deep_agent
from langchain_daytona import DaytonaSandbox

# 创建沙箱实例
sandbox = Daytona().create()
backend = DaytonaSandbox(sandbox=sandbox)

agent = create_deep_agent(
    model=llm,
    backend=backend,
    system_prompt="你是一个编程助手，可以在沙箱中创建和运行代码。请用中文回复。",
)

try:
    task = "创建一个 hello.py 文件，写入打印 Hello World 的代码，然后运行它"
    result = agent.invoke({"messages": task})
    print(result["messages"][-1].content)
finally:
    # 沙箱用完必须关闭，否则会持续计费
    sandbox.stop()
```

执行过程中，Agent 会依次调用：

1. `write_file` — 在沙箱中创建 `hello.py`
2. `execute` — 在沙箱中执行 `python hello.py`
3. 读回执行结果，向用户汇报

这里有一个前两个后端都没有的工具：`execute`。它是 SandboxBackend 的核心能力，可以在沙箱中执行任意 shell 命令。StateBackend 和 FilesystemBackend 只有文件读写工具，没有命令执行能力。

**SandboxBackend 的两种使用模式**：

- **Sandbox as Tool**（推荐）：Agent 在你的环境中运行，需要执行代码时调用沙箱工具。API Key 保留在你的环境中，不进入沙箱，更安全
- **Agent in Sandbox**：Agent 本身运行在沙箱内部，通过网络与外部通信。需要构建 Docker 镜像，部署成本更高

教程示例都使用 Sandbox as Tool 模式。

**沙箱的文件传输**：除了 Agent 通过工具在沙箱内操作文件，你的应用代码也可以主动上传/下载文件：

```python
# 上传文件到沙箱（在 Agent 运行前准备环境）
backend.upload_files([
    ("/src/main.py", b"print('Hello from host')"),
    ("/requirements.txt", b"flask\nrequests\n"),
])

# 从沙箱下载文件（在 Agent 运行后获取产物）
results = backend.download_files(["/output/report.pdf"])
for r in results:
    if r.content is not None:
        print(f"下载成功: {r.path}, 大小: {len(r.content)} bytes")
```

`upload_files` 用于在 Agent 运行前把项目代码、配置文件等推到沙箱里；`download_files` 用于在 Agent 完成后把生成的代码、报告等拿回来。

**SandboxBackend 与前两个后端的对比**：

| 特性 | StateBackend | FilesystemBackend | SandboxBackend |
|------|-------------|-------------------|----------------|
| 存储位置 | 内存 | 本机磁盘 | 远程沙箱 |
| 命令执行 | 不支持 | 不支持 | `execute` 工具 |
| 安全性 | 最高 | 中等 | 高（沙箱隔离） |
| 需要额外服务 | 否 | 否 | 是（沙箱提供商） |
| 适用场景 | 对话、预览 | 本地开发 | 代码执行、不可信代码 |

**沙箱生命周期**：沙箱是远程资源，创建后持续计费直到关闭。常见的两种管理方式：

- **线程级别**（Thread-scoped）：每个对话线程一个沙箱，对话结束就销毁
- **助手级别**（Assistant-scoped）：同一个助手共享一个沙箱，跨对话保持文件和已安装的包

实际使用中，建议配合 `try/finally` 确保沙箱一定被关闭，避免资源泄漏。

## 文件编辑

Agent 除了能读写文件，还内置了 `edit_file` 工具，支持对已有文件的编辑操作。它能做这些事：

- 替换指定行的内容
- 查找并替换特定文本
- 在文件末尾追加内容
- 删除指定行

看一个实际例子——先创建一个配置文件，再让 Agent 修改其中的值：

```python
agent = create_deep_agent(
    model=llm,
    backend=StateBackend(),
    system_prompt="你是一个代码助手。请用中文回复。修改文件时保持格式整洁。",
)

# 第一步：创建文件
task1 = "请在 config.py 中写入以下内容:\n\ntitle = 'My App'\nversion = '1.0.0'\ndebug = True\nhost = 'localhost'\nport = 8000"
result1 = agent.invoke({"messages": task1})
```

Agent 调用 `write_file` 创建了 `config.py`。接着让它修改：

```python
# 第二步：编辑文件
task2 = "请修改 config.py 文件: 把 debug 改为 False，把 port 改为 9000"
result2 = agent.invoke({"messages": task2})
```

Agent 会调用 `edit_file` 工具，找到 `debug = True` 替换为 `debug = False`，找到 `port = 8000` 替换为 `port = 9000`。

查看编辑后的文件：

```python
for path, file_info in result2["files"].items():
    if isinstance(file_info, dict):
        content = file_info.get("content", str(file_info))
    else:
        content = str(file_info)
    print(f"{path}:")
    for line in str(content).split("\n"):
        print(f"  {line}")
```

输出：

```
config.py:
  title = 'My App'
  version = '1.0.0'
  debug = False
  host = 'localhost'
  port = 9000
```

`debug` 和 `port` 的值已经被正确修改。`edit_file` 的好处是精确修改——不需要重写整个文件，只需要告诉 Agent "把什么改成什么"，Agent 会找到对应的位置做替换。

## 文件权限控制

FilesystemBackend 让 Agent 能访问真实文件系统，但并不是所有文件都应该让 Agent 碰。Deep Agent 提供了 `FilesystemPermission` 来精细控制 Agent 的文件访问权限。

先准备两个文件——一个公开的，一个机密的：

```python
temp_dir = tempfile.mkdtemp(prefix="deepagents_perm_")

with open(os.path.join(temp_dir, "public.txt"), "w") as f:
    f.write("公开信息")
with open(os.path.join(temp_dir, "secret.txt"), "w") as f:
    f.write("机密信息")
```

配置权限规则：

```python
from deepagents import FilesystemPermission

permissions = [
    # 允许读写 public 开头的文件
    FilesystemPermission(
        operations=["read", "write"],
        paths=["/public*"],
        mode="allow",
    ),
    # 禁止访问 secret 开头的文件
    FilesystemPermission(
        operations=["read", "write"],
        paths=["/secret*"],
        mode="deny",
    ),
]
```

把权限规则传给 Agent：

```python
agent = create_deep_agent(
    model=llm,
    backend=FilesystemBackend(root_dir=temp_dir, virtual_mode=True),
    permissions=permissions,
    system_prompt="你是一个助手。请用中文回复。",
)
```

测试一下——让 Agent 同时读取两个文件：

```python
task = "请读取 public.txt 和 secret.txt 的内容"
result = agent.invoke({"messages": task})

print(result["messages"][-1].content)
```

Agent 的回复大概是这样的：

```
- **public.txt** 读取成功：内容为 "公开信息"
- **secret.txt** 读取失败：权限被拒绝，无法访问该文件
```

Agent 读取 `public.txt` 成功，但读取 `secret.txt` 时被权限规则拦截了。

**权限规则的三个字段**：

| 字段 | 可选值 | 说明 |
|------|--------|------|
| operations | read, write | 控制哪些操作 |
| paths | 通配符模式 | 匹配文件路径（支持 `*` 通配符） |
| mode | allow, deny | 允许还是拒绝 |

规则匹配遵循两个原则：

1. **按声明顺序匹配**：列表中的规则从上到下逐条检查，第一条匹配的规则生效
2. **默认允许**：如果没有规则匹配到某个文件，默认允许访问

所以写规则时，"允许"规则要放在前面，"拒绝"规则放在后面。如果顺序反过来，先匹配到 `deny secret*` 的文件会被拒绝，但 `allow public*` 仍然正常。这两个规则没有冲突，顺序不影响结果。但如果出现路径重叠的情况（比如一条规则匹配 `src/*`，另一条匹配 `src/secret/*`），就需要把更具体的规则放前面。

## 安全实践

三种后端代表了三个安全等级：

**第一级：StateBackend（最安全）**

文件完全在内存中，Agent 碰不到真实文件系统。适合 Web 服务、对话场景，零风险。

**第二级：FilesystemBackend + 权限控制（中等安全）**

Agent 能访问真实文件，但通过 `virtual_mode` 和 `permissions` 限制了范围。适合本地开发工具、CLI 助手。需要注意：

- Agent 可以访问所有可读的文件，包括 `.env` 等敏感文件（除非通过 permissions 拒绝）
- 建议敏感操作启用 `interrupt_on`（Human-in-the-Loop 中断确认）

**第三级：SandboxBackend（隔离执行）**

在远程沙箱中执行文件操作和命令，与本机完全隔离。适合运行不可信代码、多租户 Web 服务。前面已经展示了 SandboxBackend 的使用方式。

一个核心原则：**安全边界在工具和沙箱层面实施，不要依赖模型自我约束。** LLM 可能会"想"去读 `.env` 文件，只要工具层面禁止了，它就做不到。反之，如果工具层面允许了但提示词里写了"不要读敏感文件"，LLM 有可能不遵守。

生产环境的建议：

- **本地开发**：FilesystemBackend + permissions + Human-in-the-Loop
- **Web 服务**：StateBackend 或 SandboxBackend
- **多租户环境**：SandboxBackend + 最小权限原则

## 小结

这里讲了 Deep Agent 文件系统的核心内容：

- StateBackend：文件存内存，安全无副作用，适合对话场景
- FilesystemBackend：读写真实磁盘，配合 `virtual_mode` 和 `permissions` 控制安全边界
- SandboxBackend：远程沙箱执行，支持 `execute` 命令执行和文件传输，适合不可信代码场景
- 文件编辑：内置 `edit_file` 工具，支持精确修改文件内容
- 权限控制：`FilesystemPermission` 按 `operations`、`paths`、`mode` 三个维度控制文件访问
- 安全实践：安全在工具层面实施，不要依赖 LLM 自我约束
