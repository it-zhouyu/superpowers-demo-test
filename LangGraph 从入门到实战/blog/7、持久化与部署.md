前两篇我们解决了两个关键问题：Agent 能在关键操作前暂停等人审批，多个 Agent 能分工协作完成任务。但这两个能力都依赖一个前提——状态被保存着。interrupt 暂停后能恢复，是因为检查点记住了暂停时的状态；多 Agent 之间能传递信息，是因为消息列表在内存中持续累加。

问题是：程序一关，内存全没了。下次启动，所有对话记录、检查点、Agent 协作的中间状态都不存在。对于开发和测试来说无所谓，但对于线上服务，这是不能接受的。用户关闭网页再打开，他的对话还在不在？服务器重启后，之前暂停等待审批的邮件还在不在？

这篇来解决状态的持久化存储和生产部署问题。

## MemorySaver：内存级持久化

回顾一下之前一直在用的 `MemorySaver`：

```python
from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()
app = build_chatbot(checkpointer)
```

MemorySaver 的特点：

- 数据存在 Python 进程的内存中
- 同一个进程内，多次 `invoke()` 共享状态
- 进程退出后数据丢失
- 不需要额外安装数据库或配置连接

它解决的问题是"同一个进程内的跨调用状态保持"。比如：

```python
config = {"configurable": {"thread_id": "user-001"}}

# 第一轮
app.invoke({"messages": [HumanMessage(content="我叫小明")]}, config)

# 第二轮 — 记得上一轮的内容
result = app.invoke({"messages": [HumanMessage(content="我叫什么？")]}, config)
# AI 回答: 你叫小明
```

只要程序还在运行，两个 invoke 之间的状态就被 MemorySaver 保持住了。开发阶段用它完全够。

## thread_id：对话隔离

多用户场景下，不同用户的对话必须隔离。`thread_id` 就是隔离的依据：

```python
config_alice = {"configurable": {"thread_id": "user-alice"}}
config_bob = {"configurable": {"thread_id": "user-bob"}}

# Alice 的对话
app.invoke({"messages": [HumanMessage(content="我住在上海")]}, config_alice)

# Bob 的对话
app.invoke({"messages": [HumanMessage(content="我住在北京")]}, config_bob)

# 回到 Alice — 她的上下文还在
result = app.invoke({"messages": [HumanMessage(content="我住哪？")]}, config_alice)
# AI 回答: 你住在上海
```

每个 thread_id 对应一条独立的状态链。换 thread_id 就是全新的对话。在 Web 服务中，通常的做法是用用户 ID + 会话 ID 作为 thread_id，确保每个用户的不同会话互不干扰。

## get_state() 与 get_state_history()：状态检查

前面用 `get_state()` 查看中断信息。它的用途不限于 interrupt，任何带 checkpointer 的图都能用它查看状态：

```python
state = app.get_state(config)

# 当前状态中的消息
for msg in state.values.get("messages", []):
    role = "Human" if isinstance(msg, HumanMessage) else "AI"
    print(f"[{role}] {msg.content[:80]}")

# 待执行的节点（空元组表示已完成）
print(f"next: {state.next}")

# 检查点创建时间
print(f"created_at: {state.created_at}")
```

`get_state_history()` 返回完整的检查点历史，每次 `invoke()` 都会生成一个：

```python
count = 0
for hist_state in app.get_state_history(config):
    count += 1
    msg_count = len(hist_state.values.get("messages", []))
    print(f"状态 {count}: next={hist_state.next}, 消息数={msg_count}")
    if count >= 10:
        break
```

输出类似：

```
状态 1: next=(), 消息数=6     <- 最新状态（已完成）
状态 2: next=('chatbot',), 消息数=5
状态 3: next=('chatbot',), 消息数=3
状态 4: next=('chatbot',), 消息数=1
状态 5: next=(), 消息数=0     <- 最初的空状态
```

检查点历史有几个用途：

- **调试**：回溯 Agent 在哪一步做了什么决策
- **时间旅行**：可以回到任意历史状态重新执行（通过 `parent_config`）
- **审计**：记录完整的执行过程

## update_state()：手动修改状态

除了自动保存，LangGraph 还允许手动修改状态。这在几种场景下很有用：

- 用户纠正了之前的信息（"我说的不是上海，是浦东"）
- 管理员需要注入额外的上下文
- 测试时模拟特定状态

```python
# 注入一条补充信息
app.update_state(
    config,
    {"messages": [HumanMessage(content="补充信息：我说的上海是指浦东新区")]},
    as_node="chatbot",
)
```

`as_node` 参数指定"这条更新假装是由哪个节点产生的"。这在图的后续执行中会影响状态的元信息（比如 `created_at`、`parent_config`），但不会真的重新执行那个节点。

修改完状态后继续对话，Agent 会看到新注入的信息：

```python
result = app.invoke(
    {"messages": [HumanMessage(content="基于补充的信息再推荐一下")]},
    config,
)
```

状态修改的几个注意点：

| 参数 | 说明 |
|------|------|
| `config` | 指定要修改哪个 thread 的状态 |
| `values` | 要合并到状态中的新值（部分更新，不是替换） |
| `as_node` | 假装更新来自哪个节点，影响元信息 |

## Checkpointer 后端选择

MemorySaver 是开发用的，生产环境需要真正的持久化存储。LangGraph 提供了多种 Checkpointer 后端：

| 后端 | 存储介质 | 适用场景 | 安装方式 |
|------|----------|----------|----------|
| `MemorySaver` | 内存 | 开发和测试 | 内置，无需额外安装 |
| `SqliteSaver` | SQLite 文件 | 单机轻量级部署 | `pip install langgraph-checkpoint-sqlite` |
| `AsyncPostgresSaver` | PostgreSQL | 生产环境推荐 | `pip install langgraph-checkpoint-postgres` |

SqliteSaver 的用法：

```python
from langgraph.checkpoint.sqlite import SqliteSaver

# 数据存在 chat.db 文件中
with SqliteSaver.from_conn_string("chat.db") as checkpointer:
    app = build_chatbot(checkpointer)
    # 正常使用
```

PostgreSQL 的用法：

```python
from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver

async with AsyncPostgresSaver.from_conn_string("postgresql://...") as checkpointer:
    await checkpointer.setup()  # 首次使用需要建表
    app = build_chatbot(checkpointer)
    # 正常使用
```

选择依据：本地开发和单元测试用 MemorySaver，单机小项目用 SqliteSaver，线上服务用 PostgreSQL。三者提供完全相同的 API，只是存储介质不同，切换时不需要改业务代码。

## 部署概述

开发完的 LangGraph 应用，最终要部署成服务。这里概述几种部署方式。

### 本地开发：langgraph dev

LangGraph CLI 提供了本地开发服务器：

```bash
pip install langgraph-cli
langgraph dev
```

它会启动一个本地 HTTP 服务，支持热重载（代码改了自动生效），自带 Web UI 用于测试和调试。适合开发阶段快速迭代。

### 云端部署：LangGraph Cloud

```bash
langgraph deploy
```

把图部署到 LangGraph Cloud（LangChain 官方的托管服务）。自动提供 API 接口、状态管理、并发处理、水平扩展。适合不想自己运维的场景。

### 自托管部署

自己搭建服务，通常的架构：

- LangGraph 应用运行在 Docker 容器中
- PostgreSQL 作为 Checkpointer 后端
- 用 FastAPI 或 LangGraph Server 暴露 API
- 用 LangSmith 做监控和追踪

生产环境的关键考虑：

| 维度 | 关注点 |
|------|--------|
| 持久化 | PostgreSQL Checkpointer，数据不丢 |
| 并发 | 不同 thread_id 的状态隔离，支持多用户同时使用 |
| 监控 | LangSmith 集成，追踪每次 Agent 执行的步骤和耗时 |
| 扩展 | LLM API 调用是瓶颈，需要考虑限流和重试 |
| 安全 | 工具执行的权限控制，interrupt 机制保护危险操作 |

部署的具体实现超出了本系列的范围。核心要点：先把图在本地开发好，然后用合适的 Checkpointer 做持久化，最后选择部署方式（云端或自托管）。

## LangGraph 系列回顾

前面把 LangGraph 的核心能力走了一遍：

| 主题 | 解决的问题 |
|------|-----------|
| 初识 LangGraph | LLM 调用是线性的，需要图编排 |
| 状态管理 | 图节点之间需要共享和传递数据 |
| 条件路由与循环 | 图需要根据运行时条件走不同路径 |
| ReAct Agent | Agent 需要自主推理和工具循环 |
| 人机交互与检查点 | 关键操作需要人工审批 |
| 多 Agent 协作 | 单 Agent 能力有限，需要专家分工 |
| 持久化与部署 | 生产环境需要持久化存储 |

核心线索是层层递进的：

- 图编排解决"多步骤流程"问题
- 状态管理解决"步骤间数据传递"问题
- 条件路由解决"流程分支"问题
- ReAct 循环解决"自主推理"问题
- interrupt 解决"人工介入"问题
- 多 Agent 解决"专家分工"问题
- 持久化解决"生产可用"问题

```python
"""
LangGraph 教程 - Step 7: Persistence（持久化）

演示状态持久化与状态管理：
1. MemorySaver 基本用法 - 跨调用保持状态
2. 多线程隔离 - 不同对话互不干扰
3. 状态检查 - get_state / get_state_history
4. 状态修改 - update_state
5. 部署方式概述
"""

import uuid
from typing import TypedDict, Annotated

from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.checkpoint.memory import MemorySaver
from langchain_core.messages import HumanMessage, AIMessage, BaseMessage
from langchain_deepseek import ChatDeepSeek
from config import DEEPSEEK_API_KEY, DEEPSEEK_MODEL, DEEPSEEK_BASE_URL

# ============================================================
# 定义状态和 LLM
# ============================================================

class ChatState(TypedDict):
    """聊天状态"""
    messages: Annotated[list[BaseMessage], add_messages]

llm = ChatDeepSeek(
    model=DEEPSEEK_MODEL,
    api_key=DEEPSEEK_API_KEY,
    api_base=DEEPSEEK_BASE_URL,
)

def chatbot_node(state: ChatState) -> dict:
    """聊天节点"""
    response = llm.invoke(state["messages"])
    return {"messages": [response]}

def build_chatbot(checkpointer=None):
    """构建聊天机器人图"""
    graph = StateGraph(ChatState)
    graph.add_node("chatbot", chatbot_node)
    graph.add_edge(START, "chatbot")
    graph.add_edge("chatbot", END)
    return graph.compile(checkpointer=checkpointer)

# ============================================================
# Demo 1: MemorySaver 基本用法
# ============================================================

def demo1_memory_saver():
    """Demo 1: 使用 MemorySaver 保持跨调用的状态"""
    print("=" * 50)
    print("Demo 1: MemorySaver 基本用法")
    print("=" * 50)

    checkpointer = MemorySaver()
    app = build_chatbot(checkpointer)

    thread_id = f"chat-{uuid.uuid4().hex[:8]}"
    config = {"configurable": {"thread_id": thread_id}}

    # 第一轮对话
    print(f"\n--- 第一轮（thread_id={thread_id}）---")
    result = app.invoke(
        {"messages": [HumanMessage(content="我叫小明，记住我的名字。")]},
        config,
    )
    print(f"AI: {result['messages'][-1].content}")

    # 第二轮对话：验证记忆保持
    print(f"\n--- 第二轮（同 thread_id）---")
    result = app.invoke(
        {"messages": [HumanMessage(content="我叫什么名字？")]},
        config,
    )
    print(f"AI: {result['messages'][-1].content}")

    # 第三轮对话：继续对话
    print(f"\n--- 第三轮（同 thread_id）---")
    result = app.invoke(
        {"messages": [HumanMessage(content="给我推荐一个 Python 学习资源。")]},
        config,
    )
    print(f"AI: {result['messages'][-1].content}")

    # 对比：没有 checkpointer 的情况
    print(f"\n--- 对比: 无 checkpointer ---")
    app_no_memory = build_chatbot()
    result = app_no_memory.invoke(
        {"messages": [HumanMessage(content="我叫什么名字？")]}
    )
    print(f"AI（无记忆）: {result['messages'][-1].content}")

def demo2_multiple_threads():

def demo3_state_inspection():
    """Demo 3: 使用 get_state 和 get_state_history 检查状态"""
    print("\n" + "=" * 50)
    print("Demo 3: 状态检查")
    print("=" * 50)

    checkpointer = MemorySaver()
    app = build_chatbot(checkpointer)

    thread_id = f"inspect-{uuid.uuid4().hex[:8]}"
    config = {"configurable": {"thread_id": thread_id}}

    # 执行几轮对话
    print(f"\n--- 执行 3 轮对话（thread_id={thread_id}）---")
    app.invoke({"messages": [HumanMessage(content="第一轮：你好")]}, config)
    app.invoke({"messages": [HumanMessage(content="第二轮：介绍一下自己")]}, config)
    result = app.invoke({"messages": [HumanMessage(content="第三轮：今天天气怎么样")]}, config)

    # 查看当前状态
    print(f"\n--- get_state(): 当前状态 ---")
    state = app.get_state(config)
    print(f"待执行节点: {state.next}")
    print(f"消息总数: {len(state.values.get('messages', []))}")
    print(f"创建时间: {state.created_at}")
    print(f"父配置: {state.parent_config}")

    # 查看所有消息
    print(f"\n--- 当前状态中的消息 ---")
    for i, msg in enumerate(state.values.get("messages", [])):
        role = "Human" if isinstance(msg, HumanMessage) else "AI"
        content = msg.content[:80] if len(msg.content) > 80 else msg.content
        print(f"  [{i}] {role}: {content}")

    # 查看状态历史
    print(f"\n--- get_state_history(): 状态历史 ---")
    history_count = 0
    for hist_state in app.get_state_history(config):
        history_count += 1
        msg_count = len(hist_state.values.get("messages", []))
        print(f"  历史 {history_count}: "
              f"next={hist_state.next}, "
              f"消息数={msg_count}, "
              f"时间={hist_state.created_at}")
        if history_count >= 10:
            print(f"  ... (只显示前 10 条)")
            break

    # 统计总历史数
    total = sum(1 for _ in app.get_state_history(config))
    print(f"\n  共 {total} 个历史状态")
```

搭建 Agent 的基础设施都掌握了——状态、图、工具、人机交互、多 Agent、持久化。但每次都要自己写图、加节点、配工具。DeepAgents 提供了开箱即用的 Agent 套件，用声明式的方式定义 Agent，不用手写图编排代码。那是下一个教程系列的内容。
