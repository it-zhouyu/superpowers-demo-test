⏺ /branch 命令的作用

  /branch 用于在当前对话的节点上创建一份完整的对话副本，然后将你切换到这个新分支
  中继续工作，原始会话保持不变

  简单理解就是"分叉"当前对话——你可以在分支中尝试不同的方向，同时不丢失已有的对话
  路径

  基本用法

  /branch try-streaming-approach

  name 参数是可选的，给分支会话起个名字方便后续在 session picker 中查找

  是否继承当前会话的历史

  是的，完全继承。 /branch
  创建的是当前对话的完整拷贝，包含到此刻为止的所有对话历史、上下文和状态

  执行后，终端会打印两个 session ID：
  - 你所在的新分支的 ID
  - 原始会话的 ID

  你可以随时通过 /resume 加原始会话的 ID 或名称切回去

  注意事项

  - 权限不继承：通过 "allow for this
  session"（仅本次会话允许）批准的权限不会延续到新分支，需要在分支中重新批准
  - 原始会话不受影响：原始会话完整保留，在 session picker 中仍然可见和可恢复
  - 分组显示：通过 /branch、/rewind 或 --fork-session 创建的分叉会话，在 session
  picker 中会归组显示在根会话下面，按 → 键展开查看

  /branch 与 /fork 的区别

  从 v2.1.161 开始两者行为不同：

  ┌─────────┬────────────────────────────────────────────────────────────────┐
  │  命令   │                              行为                              │
  ├─────────┼────────────────────────────────────────────────────────────────┤
  │ /branch │ 把你本人切换到对话副本中继续工作                               │
  ├─────────┼────────────────────────────────────────────────────────────────┤
  │ /fork   │ 生成一个后台 subagent                                          │
  │         │ 继承对话并处理任务，而你继续留在当前会话中                     │
  └─────────┴────────────────────────────────────────────────────────────────┘

  在 v2.1.161 之前的版本中，/fork 是 /branch 的别名

  ┌────────────┬────────────────────┬────────────────────────────────┐
  │            │      /branch       │             /fork              │
  ├────────────┼────────────────────┼────────────────────────────────┤
  │ 谁继续工作 │ 你本人切换到副本中 │ 后台 subagent 继承对话处理任务 │
  ├────────────┼────────────────────┼────────────────────────────────┤

  则是另一种模式，会把你直接切换到新的对话副本中。

❯ 为什么不行                                                                    

⏺ /security-review 报错的原因是它依赖
  origin/HEAD（远程仓库的默认分支引用），但这个项目没有配置远程仓库，所以 Git
  找不到 origin/HEAD。

  从错误信息看：

  fatal: ambiguous argument 'origin/HEAD...': unknown revision or path not in
  the working tree.

  你刚才执行了 git init 重新初始化了仓库，但这个仓库是全新的，没有 origin
  远程地址。/security-review 需要对比
  origin/HEAD...（即远程默认分支与当前分支的差异）来确定要审查哪些文件。

  解决办法

  添加一个远程仓库地址即可：

  git remote add origin <你的远程仓库URL>
  git fetch origin

  或者，如果你只是想在本地做一个初始提交然后再审查：

  git commit -m "初始提交"
  # 然后添加远程仓库
  git remote add origin <远程仓库URL>
  git push -u origin main

  之后再执行 /security-review 就不会报错了。
